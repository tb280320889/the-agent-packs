## Summary

本 phase 的本质不是“写新功能”，而是把当前 brownfield 仓库从“可运行但散乱”升级为“可扩展、可审阅、可协作”的工程化结构基线。结合 AIDP 与 `.planning/codebase/*` 现状，仓库当前已有明确技术分层（`cmd/`、`internal/`、`blueprint/`、`workflow-packages/`、`docs/AIDP/`），因此本阶段应采用“语义归位 + 提交治理”策略，而不是大规模逻辑改写。

规划重点应放在三件事：第一，先冻结目录目标模型与根目录白名单；第二，执行“移动/归档优先、删除延后”的安全重排；第三，用可回滚的分批 commit 把风险切碎。这样可以在激进重排前提下，保持审阅可读性与历史可追踪性，同时不破坏 AIDP 的“业务语义真相源”定位。

从 Git 官方文档看，`git mv` 会更新 index 但仍需提交；`git diff` 支持工作区/暂存区/分支差异比对，且可用于大重排场景的 rename/summary 审阅；`git commit` 明确建议以简洁标题 + 详细正文组织提交说明。结合本仓库约束，建议采用“按变更意图分批，不按时间分批”的策略：结构迁移、引用修复、文档映射、归档清单、验证结果分别提交。

**Primary recommendation:** 先确定“目标目录模型 + 根目录白名单 + 分批提交切片规则”，再执行激进迁移；始终坚持“先归档后删除、删除需确认、每批可独立审阅回滚”。

|---|---|---|---|
| Git (`git mv`, `git diff`, `git commit`) | 2.53.0 docs baseline | 文件迁移、差异审阅、分批提交 | 官方原生命令覆盖重构提交流程，语义清晰且可追踪 |
| Go toolchain | 1.25 | 重构后编译/测试验收 | 本仓库主语言与既有测试基线 |
| AIDP + `.planning` 工件体系 | local AIDP 1.1.0 | 语义约束与任务真相分层 | 已被冻结为本项目入口与协同规则 |

### Supporting
| Tool/Artifact | Version | Purpose | When to Use |
|---|---|---|---|
| `go test ./...` | Go std | 重构后回归验证 | 每一批结构迁移后、最终提交前 |
| `git diff --name-status -M` | Git builtin | 审阅 rename/move 与状态变化 | 每一批 commit 前核对迁移正确性 |
| `.planning/codebase/*` | 2026-03-16 snapshot | 作为 brownfield 结构事实基线 | 制定“现状 -> 目标”迁移映射时 |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|---|---|---|
| `git mv` 逐步迁移 | 文件系统直接移动 | 可能丢失清晰 rename 语义，不利于审阅与 blame 追踪 |
| 分批提交 | 一次性大提交 | 大提交审阅成本高、回滚粒度过粗、风险定位困难 |
| 先归档后删除 | 直接删除 | 无审阅缓冲，误删风险高，不符合锁定决策 |

**Execution command baseline:**
```bash

```

## Architecture Patterns

### Recommended Project Structure
```text
[repo-root]/
├── AGENTS.md                     # 唯一全局入口约束
├── go.mod / go.sum               # Go 构建基线
├── cmd/                          # 入口程序
├── internal/                     # 核心实现
├── blueprint/                    # 领域知识源与索引
├── workflow-packages/            # package 清单与注册表
├── tests/                        # 测试
├── fixtures/                     # 固定样例
├── docs/
│   ├── AIDP/                     # 语义真相源（当前主入口）
│   └── 改造计划v1/               # 历史来源（非默认入口）
└── .planning/                    # GSD 任务真相与执行工件
```

### Pattern 1: Two-Track Refactor (结构迁移轨 + 语义对齐轨)
**What:** 把重构拆为“目录迁移”与“文档/引用对齐”两条并行逻辑轨，但在提交上按可审阅性串行落地。
**When to use:** brownfield 体量可控、允许激进重排、但必须保持协作可读性。
**Example:**
```bash
# 1) 仅迁移目录/文件，不混入逻辑改动
git mv old_path new_path

# 2) 单独提交引用修复与路径更新
git add <updated-references>
git commit -m "refactor(paths): align references after repository re-layout"
```
Source: https://git-scm.com/docs/git-mv, https://git-scm.com/docs/git-commit

### Pattern 2: Archive-First Cleanup
**What:** 所有“疑似废弃/散乱”资产先进入归档区并带清单，再等待确认删除。
**When to use:** 需要安全清理、且用户已锁定“先归档后删除”策略。
**Example:**
```text
docs/archive/
  ├── 2026-03-phase-01.1/
  │   ├── ARCHIVE-LIST.md
  │   └── <moved legacy assets>
```

### Pattern 3: Intent-Sliced Commits
**What:** 提交按“意图类别”切片，而非按作者操作顺序切片。
**When to use:** 大规模目录重排，且需要提升 reviewer 理解速度。
**Recommended slices:**
1. `refactor(structure):` 目录迁移与根目录收敛
2. `refactor(paths):` import/路径/脚本引用修复
3. `docs(aidp):` 文档映射与迁移说明
4. `chore(archive):` 归档清单与风险分级
5. `test/verify:` 回归验证与证据回写

### Anti-Patterns to Avoid
- **一把梭大提交:** 把移动、重命名、逻辑修改、文档改写混在一个 commit。
- **边删边改无归档:** 直接删除无法确认价值的资产，破坏审阅链路。
- **以 `.planning` 反向覆盖 AIDP 语义:** 违反“GSD 管任务、AIDP 管语义”的边界。

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| 文件迁移追踪 | 自写迁移脚本维护 rename 映射 | `git mv` + `git diff --name-status -M` | Git 原生已有可审阅 rename 语义 |
| 差异统计 | 手工整理“改了什么”列表 | `git diff`/`git status` 标准输出 | 减少漏项与主观偏差 |
| 提交格式治理 | 临时口头约定 | 统一 commit 模板（标题+正文） | 避免后续审阅不可读 |
| 任务语义解释 | 在 PR/对话中临时说明 | AIDP runtime + `.planning` 工件回写 | 保持真相源可追踪 |

**Key insight:** 本 phase 的复杂度不在代码算法，而在“变更可治理性”；优先复用 Git 与既有 AIDP/GSD 工件能力，避免额外自定义流程工具化。

## Common Pitfalls

### Pitfall 1: 根目录白名单未先冻结
**What goes wrong:** 迁移过程反复改目标，导致多次挪动和冲突提交。
**Why it happens:** 先动手后建模，没有先定义顶层语义边界。
**How to avoid:** 先产出“根目录保留项白名单 + 目标树”，再执行迁移。
**Warning signs:** 同一目录在连续 commit 中多次改名/改层级。

### Pitfall 2: 把结构迁移和逻辑修改混在同一批
**What goes wrong:** Reviewer 无法判断行为变更是否由路径迁移引起。
**Why it happens:** 追求一次做完，忽略审阅粒度。
**How to avoid:** 先纯迁移提交，再单独修复引用和逻辑。
**Warning signs:** 单个 commit 同时出现大量 rename + 业务逻辑 diff。

### Pitfall 3: 忽略 AIDP/GSD 双真相边界
**What goes wrong:** 任务状态、语义结论、代码事实互相覆盖，后续 agent 接力失败。
**Why it happens:** 没按 `docs/AIDP/adapters/gsd/*` 协同规则回写。
**How to avoid:** 按“任务状态写 `.planning`，语义决策写 `docs/AIDP/runtime`”执行。
**Warning signs:** 结论只存在 commit message 或会话，不在运行态工件。

### Pitfall 4: 直接删除散乱资产
**What goes wrong:** 误删后无法快速追溯、恢复成本高。
**Why it happens:** 将“清理”误解为“立刻删除”。
**How to avoid:** 强制归档先行，删除动作独立待确认批次。
**Warning signs:** 出现“无法确认删除依据/责任人/恢复路径”的审阅反馈。

## Open Questions

1. **归档落位是否采用 `docs/archive/` 还是 `.planning/archive/`**
   - What we know: 必须先归档，后删除。
   - What is unclear: 归档更偏“长期文档资产”还是“阶段执行工件”。
   - Recommendation: 默认 `docs/archive/`（长期可读），并在 `.planning/phases/01.1/` 放索引链接。

2. **分批 commit 的最小粒度阈值**
   - What we know: 必须分批，强调可审阅与可追踪。
   - What is unclear: 单批文件数量/变更行数阈值。
   - Recommendation: 以“单批单意图、reviewer 10 分钟可读完”为实操阈值。

3. **是否在 phase 内引入自动化目录检查脚本**
   - What we know: 本 phase 目标是结构治理，不是功能扩展。
   - What is unclear: 是否允许增加轻量检查脚本作为护栏。
   - Recommendation: 先不引入新工具，优先使用现有 Git + `go test` + 文档清单；若痛点明显再进入后续 phase。
