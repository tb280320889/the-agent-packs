# Phase 01.1: brownfield-git-commit - Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

在当前 brownfield 项目范围内，进行一次“大检查 + 大重构”：
- 按语义与领域完成目录重组，消除根目录与各处散乱资产
- 保持本阶段聚焦于“结构与组织治理”，不扩展新功能能力
- 完成可审阅、可协作的分批 git commit 提交

本阶段讨论只定义“如何重构与提交”，不新增独立能力。

</domain>

<decisions>
## Implementation Decisions

### 目录分层与语义边界
- 目录组织采用“先技术类型、后领域细分”的主原则。
- 根目录应尽量只保留必要入口与必要配置，不保留无关目录。
- 具体顶层分类、白名单、命名细则由 Claude 基于成熟开源团队最佳实践与项目实际落地。

### 重构力度与兼容策略
- 采用激进重排策略（项目体量可控，允许较大结构调整）。
- 兼容与迁移细节（如是否生成迁移映射文档、过渡层策略、验证门槛）由 Claude 按最佳实践决策。

### 散乱内容清理规则
- 清理策略固定为：先归档，后删除。
- 删除动作必须在用户确认后执行。
- 归档位置、优先级、审阅清单形式、删除触发流程由 Claude 按最佳实践制定。

### 最终提交与变更表达
- 采用分批 commit，不做一次性大提交。
- commit message 按工程团队协作最佳实践编写，强调可审阅与可追踪。
- 分批粒度与 message 具体格式由 Claude 按最佳实践决策。

### Claude's Discretion
- 顶层目录分组与命名规范细节（在“先技术类型后领域”约束内）。
- 根目录保留项白名单细则（遵循成熟开源项目实践）。
- 迁移说明形式、重构节奏与验证门槛选择。
- 归档清单结构与风险分级呈现方式。
- commit 分批策略（按可审阅性最优）与具体 message 规范。

</decisions>

<specifics>
## Specific Ideas

- 用户强调重构应遵循“开发工程师最佳架构实践”，避免目录散乱。
- 用户明确接受激进重排（代码量不大，重构窗口可控）。
- 用户要求清理采取“先归档、确认后删除”的安全治理策略。
- 用户要求提交采用“团队协作友好”的分批提交与规范化 message。

</specifics>
