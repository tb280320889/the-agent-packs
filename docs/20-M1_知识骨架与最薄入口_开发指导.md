# 20 M1 知识骨架与最薄入口 开发指导

## 作用域
本里程碑只负责：
- 建立最小目录骨架
- 建立首批 blueprint nodes
- 实现最薄 activation entry
- 实现最薄 blueprint compiler / query
- 让 route 与最小 bundle 真正可跑

本里程碑不负责：
- 完整 pack 深实现
- 大规模 artifact 生成
- 复杂 CLI / 服务编排
- 多领域扩展

## 本里程碑唯一目标
让系统第一次具备“能选主 pack、能带最小上下文、能诚实返回上下文不足”的能力。

## 必读文档
- `01-通用开发方法论与里程碑解耦规则.md`
- `21-M1_上下文_Blueprint_命名_目录.md`
- `22-M1_上下文_Routing_Bundle_最薄入口.md`

## 输入前提
- M0 已冻结协议、命名、前置规则
- Activation Result / Validation Plan / route 裁决已经固定
- WXT 为固定首条领域线

## 本里程碑交付物
1. 顶层目录骨架
2. 首批 blueprint 节点
3. frontmatter 可解析
4. 最薄 activation entry
5. 最薄 blueprint compiler / query
6. 最小 context bundle 生成逻辑

## 实施步骤

### Step 1：按固定目录建仓
仅建最小目录，不做大而全：
- docs
- blueprint
- tools
- packages
- skills
- resources
- fixtures

### Step 2：落首批 blueprint 节点
必须先有：
- L0.wxt
- L1.wxt.manifest
- L1.wxt.content-script
- L1.security.permissions
- L1.release.browser-store
- L2.wxt.content-script.page-injection

### Step 3：实现 frontmatter 解析
最少要解析：
- id
- level
- domain
- subdomain
- triggers
- anti_triggers
- required_with
- entry_conditions
- stop_conditions
- recommended_artifacts
- recommended_validators

### Step 4：实现最薄 activation entry
只做：
- 接收 activation request
- 识别 target_pack / target_domain / task
- 调用最薄 route
- 返回 route result 或 activation result

### Step 5：实现最薄 query / bundle
只做：
- 找主节点
- 带上 required_with
- 输出 main_pack / main_blueprint_node / required nodes / recommended validators / recommended artifacts

### Step 6：打通“诚实失败”
对于泛任务或上下文不足任务，必须能：
- 退回 L0
- 返回上下文不足
- 推荐补充 selected_files / config_fragments
不能编造宿主信息，不能强行深路由。

## 不要做的事
- 不要实现全文检索平台
- 不要做图数据库
- 不要把 blueprint 写成百科
- 不要开始写完整 validator
- 不要跳到 wxt-background-runtime
- 不要默认索引整个宿主

## 完成定义
只有同时满足以下条件，本里程碑才算完成：
1. manifest 类任务能 route 到 wxt-manifest
2. content-script 类任务能 route 到 wxt-content-script
3. 模糊任务只能到 L0 或 partial
4. bundle 不会无节制塞入无关节点
5. route 结果可解释
6. M2 可以在此基础上直接实现 wxt-manifest

## 向 M2 的 handoff
M2 可直接依赖：
- 已稳定目录
- 已稳定 frontmatter
- 已稳定 route / bundle 最小行为
- 已稳定首批节点 id 与 required_with 关系

M2 不应重新设计：
- 路由优先级
- frontmatter 必填字段
- 目录结构
