# 10 M0 规则冻结与契约补齐 开发指导

## 作用域
本里程碑只负责：
- 冻结首期的核心规则
- 补齐总纲中已经提出但尚未完全操作化的契约缺口
- 输出能被后续里程碑直接消费的稳定定义

本里程碑不负责：
- 大规模写 pack
- 实现复杂 CLI / 服务
- 扩写大量 blueprint
- 做 validator 细节

## 本里程碑唯一目标
把系统从“原则完整”推进到“对象稳定、可开始搭建”。

## 必读文档
- `01-通用开发方法论与里程碑解耦规则.md`
- `11-M0_上下文_系统定义与硬边界.md`
- `12-M0_上下文_协议契约与缺口补齐.md`

## 输入前提
- 已有总纲主设计
- 已确认 Phase 1 主目标只有一条最小闭环
- 已确认首条领域线固定为 WXT，首个完整 pack 固定为 wxt-manifest

## 本里程碑交付物
1. 固定的命名与对象表
2. Activation Result 正式文档定义
3. Validation Plan 正式文档定义
4. 路由裁决规则
5. pack 局部 contract 扩展规则
6. breaking change 分级规则
7. Phase 1 冻结边界说明

## 实施步骤

### Step 1：冻结对象词表
至少固定：
- pack
- blueprint node
- artifact
- validator
- protocol
- schema
- handoff
- activation result
- validation plan

### Step 2：补齐 Activation Result
必须给出：
- 最小字段
- 状态枚举
- completed / partial / handoff / failed 的判定标准
- 与 artifacts / validation_results / handoff 的关系

### Step 3：补齐 Validation Plan
必须给出：
- plan 来源
- 组成字段
- required_artifacts
- required_validators
- gating_rules
- boundary_assertions

### Step 4：补齐 Routing 裁决
必须给出：
- 优先级
- tie-breaker
- 上下文不足时何时只退回 L0
- anti_trigger 的阻断效力
- 显式 target_pack 不被模糊 hint 覆盖

### Step 5：补齐局部 contract 规则
必须给出：
- 全局 schema 与 pack 局部 contracts 的边界
- 哪些可以扩展
- 哪些禁止覆盖
- 如何避免 pack 各自发明 output 结构

### Step 6：补齐 breaking change 规则
至少分成：
- 文档级非破坏
- 协议级兼容扩展
- 破坏性变更
- 冻结期间禁止变更

### Step 7：输出 handoff 给 M1
handoff 必须写清：
- M1 可以假定哪些对象已经稳定
- M1 不该再改哪些字段
- 仍存在但可接受的未决项

## 不要做的事
- 不要为了单个 pack 的便利去改全局 schema
- 不要开始写复杂运行时
- 不要先发明大量 artifact 类型
- 不要先扩第二领域线
- 不要把协议文档写成散文说明

## 完成定义
只有同时满足以下条件，本里程碑才算完成：
1. Activation Result / Validation Plan 已具备正式定义
2. route 决策已具备冲突裁决规则
3. schema / pack 局部 contracts 的边界已写清
4. breaking change 规则已写清
5. 后续 M1 不需要再次讨论这些对象的基本语义

## 向 M1 的 handoff
M1 应只消费以下输出：
- 已冻结的 frontmatter / package.yaml / naming 规则
- 已冻结的协议对象
- 已补齐的路由裁决与 contract 规则

M1 不应重新解释：
- 为什么要 bounded context
- 为什么要 handoff
- 为什么不能 activation-free
这些已经在 M0 冻结。
