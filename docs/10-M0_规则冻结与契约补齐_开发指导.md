# 10 M0 规则冻结与契约补齐 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M0 只做一件事：**把全项目的公共语言、公共对象、公共协议、公共裁决规则冻结下来。**  
它不做完整 pack，不做厚实现，不扩领域树。

## 本里程碑的目标
1. 冻结系统级术语与对象名
2. 补齐 v1 缺失或未一级化的 schema / protocol
3. 冻结 route 优先级与冲突裁决
4. 冻结全局 schema 与 pack 局部 contract 的关系
5. 冻结 breaking change 基线
6. 给 M1 提供不会漂移的输入壳层

- 01

## 默认不读
- `20+`  
理由：M0 不应被具体 pack 细节干扰。

## 输入
- 系统层母版
- 当前总纲中的协议、schema、route、handoff 思路
- v1 中已识别出的缺口

## 必须产出
1. **术语冻结清单**
2. **协议冻结清单**
3. **6 个核心对象的最小 shape**
   - activation-request
   - context-bundle
   - artifact
   - handoff-bundle
   - activation-result
   - validation-plan
4. **route 裁决规则**
5. **局部 contract 扩展规则**
6. **breaking change 分类基线**
7. **M1 可直接消费的冻结对象摘要**

## 明确不产出
- 完整 compiler
- 完整 query MCP
- 完整 workflow package
- 领域 validator 细则
- 全量测试矩阵

## 执行顺序

### 第一步：冻结对象名称
先把名称定死，不允许同物多名。  
至少冻结：
- Agent Enhancement Pack
- Activation Request
- Routing Result
- Context Bundle
- Artifact
- Validation Plan
- Validator Result
- Activation Result
- Handoff Bundle
- Blueprint Node
- Blueprint Query MCP

### 第二步：补齐缺失协议
把以下对象从“文档提到”提升为“正式一等对象”：
- Activation Result
- Validation Plan

### 第三步：冻结 route 裁决
至少写清：
- 优先级顺序
- 冲突如何裁决
- 何时退回 L0
- 何时返回 partial / failed / handoff

### 第四步：冻结全局 / 局部 contract 边界
明确：
- pack 可以局部扩展什么
- 绝对不能改什么
- 什么算 breaking change

### 第五步：给 M1 输出冻结物
M1 只应在冻结壳层上工作。  
M0 必须把可直接消费的固定对象交给 M1。

---

## 完成标准
只有当以下条件全部成立，M0 才算完成：

1. M1 不需要重新解释术语
2. M1 不需要运行时补 schema
3. route 结果具备可解释性
4. activation result / validation plan 已有稳定最小 shape
5. 局部 contract 扩展边界已经明确
6. 后续里程碑知道哪些对象不能再随意改

## handoff 给 M1 的内容
- 冻结对象名单
- 不可修改字段名单
- 路由裁决表
- 允许的局部扩展面
- 当前未解决但不阻塞 M1 的风险
