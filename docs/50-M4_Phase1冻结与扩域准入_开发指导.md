# 50 M4 Phase 1 冻结与扩域准入 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M4 负责把首期系统正式冻结，并定义第二领域线如何准入。  
M4 不再追求“多做一个功能”，而是追求 **把第一阶段方法固定成可复制骨架**。

## 本里程碑的目标
1. 冻结 Phase 1 的公共壳层与首个切片
2. 明确什么能改、什么不能改
3. 明确第二领域线进入前的准入条件
4. 输出“复制而不是重造”的扩域说明

## 本里程碑对应 bead 与建议拆分
- 主 bead：`the-agent-packs-3w0`
- 建议子任务（可在 `bd` 中拆分为 task）：
  - M4-1 冻结面盘点（输出冻结对象清单）
  - M4-2 修改纪律固化（输出 non-breaking / breaking 判定规则）
  - M4-3 准入门槛定义（输出第二领域线准入清单）
  - M4-4 复用骨架落地（输出复制步骤与输入输出契约）
  - M4-5 交接与对外边界声明（输出 snapshot / handoff / 能力边界说明）

---

## 执行顺序

### 第一步：列冻结清单
把真正稳定下来的对象列出来，而不是笼统写“基本稳定”。

### 第二步：明确可改 / 不可改
对 envelope、route、bundle、package template、validator 基线做清楚切分。

### 第三步：定义第二领域线准入
不是谁想接就接。  
必须满足准入条件。

### 第四步：固化可复用 skeleton
把“怎么复制到第二条领域线”写成清晰步骤。

### 第五步：沉淀交接与对外能力边界
把阶段事实、冻结对象、风险与下一步准入路径写入共享文档，保证下游不依赖口头上下文。

## 对外能力边界说明（M4 必须给出）
M4 完成后需要明确：
- 已证明能力：`Blueprint -> compiler -> index -> query MCP -> bundle -> workflow package -> validator -> activation result / handoff` 闭环成立。
- 未证明能力：第二领域线完整 pack 尚未落地，当前只完成准入与复用骨架。
- 禁止误解：M4 不是系统层重写窗口，不能借扩域名义改动冻结壳层。

## 验收清单（DoD）
1. 冻结对象清单可枚举，且每个对象有冻结原因。
2. 任意变更可被判定为 non-breaking 或 breaking，并有明确处理路径。
3. 第二领域线存在可执行准入检查清单，不依赖口头判断。
4. 复用步骤可由新 agent 直接执行，不需要回看历史聊天。
5. `bd` 状态、Context Snapshot、Handoff 三者可追溯一致。

---

## 完成标准
1. Phase 1 冻结面清楚
2. 下游知道如何提出 breaking change
3. 第二领域线进入条件明确
4. 复用方法清楚，不依赖口头解释

## 本里程碑绝对不要做的事
- 顺手重写系统层
- 因为想接第二领域线就先把第一阶段再松开
- 用“以后再说”回避 breaking change 治理
- 只在对话中描述冻结结论而不落地为共享文档
