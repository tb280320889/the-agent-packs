# 12 M0 上下文：协议契约与缺口补齐

## 文档类型
单里程碑所需上下文文档

## 作用域
本文件负责把系统中的公共协议、公共对象 shape、route 裁决与 contract 边界补齐为正式规则。  
它是 M0 的核心上下文文档之一。

## 本文必须回答的问题
1. 哪些协议是系统级一等对象。
2. 每个协议最小必须包含什么。
3. v1 缺的对象如何补齐。
4. route 如何裁决。
5. 全局 schema 与 pack 局部 contract 如何共存。
6. 什么属于 breaking change。

---

## 一、系统级协议与对象清单

首期至少固定以下对象：

1. Activation Request
2. Routing Result
3. Context Bundle
4. Artifact
5. Handoff Bundle
6. Validation Plan
7. Validator Result
8. Activation Result

> 其中 6 和 8 在 v1 已识别为必须补齐的一等对象。

---

## 二、最小对象 shape

### 1. Activation Request
用途：调用者 agent 对系统发起一次增强运行。

最小字段：
- `request_id`
- `task`
- `target_pack?`
- `target_domain?`
- `bounded_context`
- `context_hints?`
- `selected_files?`
- `config_fragments?`

### 2. Routing Result
用途：表达 route 的选择结论。

最小字段：
- `main_pack`
- `main_blueprint_node`
- `required_packs`
- `route_reason`
- `recommended_validators`
- `recommended_artifacts`

### 3. Context Bundle
用途：给当前 pack 或当前阶段提供最小上下文闭包。

最小字段：
- `main`
- `required`
- `execution_children`
- `deferred`
- `recommended_validators`
- `recommended_artifacts`

### 4. Artifact
用途：表达一次运行生成的单个交付物。

最小字段：
- `artifact_type`
- `path_or_inline`
- `summary`
- `producer_pack`
- `status`

### 5. Handoff Bundle
用途：表达当前 pack 到达边界后，交接给相邻 pack 的结构化对象。

最小字段：
- `from_pack`
- `to_pack`
- `reason`
- `carry_context`
- `suggested_next_steps`

### 6. Validation Plan
用途：表达当前运行计划要跑哪些 validator，以及为什么跑。

最小字段：
- `plan_id`
- `request_id`
- `main_pack`
- `validators`
- `artifacts_under_validation`
- `severity_policy`
- `plan_reason`

### 7. Validator Result
用途：表达单个 validator 的输出。

最小字段：
- `validator_name`
- `status`
- `findings`
- `repair_suggestions`
- `validated_artifacts`

### 8. Activation Result
用途：表达一次 activation 的统一结果容器。

最小字段：
- `request_id`
- `status`
- `main_pack`
- `artifacts`
- `validation_results`
- `handoff?`
- `summary`

## 三、Activation Result 状态冻结

首期只允许以下状态：
- `completed`
- `partial`
- `handoff`
- `failed`

### 解释
- `completed`：已 route、已分析、已验证、已生成 artifacts
- `partial`：已有部分结果，但上下文或边界不足
- `handoff`：当前 pack 正常到达边界，返回 handoff bundle
- `failed`：协议不满足、task 不成立、无法进入该 pack

首期不要引入更细状态。

---

## 四、Validation Plan 的冻结规则

### 目标
Validation Plan 不负责执行 validator，它只负责 **提前声明验证计划**。

### 必须具备的能力
- 指定要跑哪些 validator
- 指定验证哪些 artifact
- 指定失败时的严重级别解释
- 说明为什么需要这些 validator

### 不应该承载
- 执行日志
- 真正的 findings 结果
- pack 的自由文本推理

---

## 五、route 优先级与裁决规则

### 固定优先级
1. 显式 `target_pack`
2. 显式 `target_domain`
3. Blueprint `triggers / anti_triggers`
4. `selected_files / fragments` 的有限信号
5. `context_hints`

### 必须成立的裁决规则

#### 规则 A：显式 target_pack 永远压过模糊 hint
不能因为 hints 更“像”别的包，就覆盖显式指定 pack。

#### 规则 B：显式 target_domain 可在该域内继续 route，但不能跳域
没有更强证据时，不允许跳出显式指定 domain。

#### 规则 C：anti_triggers 可以做强排除
若主域被明确 anti_trigger 命中，应返回解释，而不是硬进包。

#### 规则 D：上下文不足时，优先退回 L0 / partial
不要在证据不足时强行深路由到 L2/L3。

#### 规则 E：route 必须可解释
至少回答：
- 为什么选中 main pack
- 为什么挂上这些 cross-cutting packs
- 为什么没有选其他候选

## 六、何时返回 partial / handoff / failed

### 返回 partial
适用于：
- 方向基本成立
- 但 bounded context 不足
- 当前只能给局部结论或建议补料

### 返回 handoff
适用于：
- 当前 pack 已完成自身边界内工作
- 下一步必须进入相邻 pack
- 需要把 carry_context 显式交接下去

### 返回 failed
适用于：
- activation request 不成立
- task 明显不属于本系统
- 关键前置条件不满足
- 明确被 protocol 或 anti_trigger 排除

---

## 七、全局 schema 与 pack 局部 contract 的边界

### 允许 pack 局部扩展的内容
- domain-specific inputs
- domain-specific artifacts
- domain-specific handoff payload
- domain-specific validator fields

### 不允许 pack 改写的内容
- 系统级 envelope 对象的顶层语义
- 顶层必填字段的含义
- 顶层状态枚举
- 顶层 handoff / validation / activation 的基线行为

### 原则
**局部 contract 只能做加法，不能改写全局 envelope。**

---

## 八、breaking change 的首期分类

### 非 breaking
- 新增可选字段
- 新增 validator 类型
- 新增 artifact 子类型
- 新增 domain-specific contract，但不改顶层 envelope

### breaking
- 改顶层字段语义
- 删除顶层字段
- 改状态枚举
- 改 route 优先级顺序
- 改 handoff 的核心语义
- 改 bundle 的最小结构

### 处理原则
breaking change 不允许在 M1/M2/M3 私自发生。  
必须进入 M4 的冻结与治理流程。

---

## 九、M0 交给 M1 的冻结摘要
M0 完成后，M1 应默认接受以下对象已冻结：
- 术语
- 协议对象名
- 顶层 envelope shape
- route 优先级
- partial / handoff / failed 的语义
- 局部 contract 扩展边界
- breaking change 基线
