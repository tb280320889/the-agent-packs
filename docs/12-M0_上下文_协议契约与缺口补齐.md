# 12 M0 上下文：协议契约与缺口补齐

## 作用域
本文只回答四类问题：
1. 总纲中提出但尚未完全落地的协议对象怎么补齐
2. 全局 schema 与 pack 局部 contracts 的边界怎么固定
3. route 冲突怎么裁决
4. 变更何时算 breaking change

## 一、补齐对象 1：Activation Result

### 目标
Activation Result 是一次 activation 的统一结果容器。  
它不是单个 artifact，而是调用者 agent 可稳定消费的总结果对象。

### 最小字段
```json
{
  "request_id": "req-001",
  "status": "completed",
  "main_pack": "wxt-manifest",
  "route": {
    "main_blueprint_node": "L1.wxt.manifest",
    "required_packs": ["security-permissions", "release-engineering"],
    "route_reason": "task 明确指向 manifest，且存在 store submission / permissions 信号"
  },
  "artifacts": [],
  "validation_results": [],
  "handoff": null,
  "summary": "Manifest 范围已完成初步分析，可直接消费主 artifact。",
  "missing_context": [],
  "recommended_next_request": null
}
```

### 状态枚举
- `completed`：主 pack 已完成本职职责，必要 artifacts 与 validations 已返回
- `partial`：已生成部分结论，但上下文不足或部分对象未能验证
- `handoff`：当前 pack 已到边界，下一步应进入别的 pack
- `failed`：请求不成立、协议不满足、无法进入本 pack

### 状态判定规则
- 有可消费 artifacts，且主问题仍属当前 pack，验证已完成或可接受 -> `completed`
- 有部分 findings，但关键上下文不足，或 validator 给出 warn/fail 需补输入 -> `partial`
- 当前主问题已转移到别的 pack，且 handoff bundle 已完整 -> `handoff`
- 缺 request_id / task，或当前任务与 pack/domain 明显不匹配 -> `failed`

## 二、补齐对象 2：Validation Plan

### 目标
Validation Plan 是“本次 activation 该如何被验证”的正式对象。

### 最小字段
```json
{
  "plan_id": "vp-001",
  "request_id": "req-001",
  "main_pack": "wxt-manifest",
  "required_artifacts": ["manifest-review.md"],
  "required_validators": [
    "validator-core-output",
    "validator-domain-wxt-manifest"
  ],
  "gating_rules": [
    "若缺少 manifest-review.md，则结果不能 completed",
    "若 domain validator fail，则结果只能为 partial 或 failed"
  ],
  "boundary_assertions": [
    "不得默认推断未提供的宿主目录结构",
    "不得把 release 主问题继续留在 wxt-manifest 内解决"
  ]
}
```

### 来源规则
Validation Plan 由三部分共同决定：
1. `package.yaml` 的 validators / artifacts / exit_criteria
2. blueprint node 的 recommended_validators / recommended_artifacts
3. 当前 route / bundle 上下文

### 作用
Validation Plan 不判断结论“是否正确”，它判断：
- 输出结构是否成立
- 必需 artifact 是否存在
- 边界是否被说明
- 是否需要 handoff
- 是否还能声明 completed

## 三、补齐 Routing 裁决规则

### 基本优先级
1. `target_pack`
2. `target_domain`
3. `task` 与 blueprint triggers / anti_triggers
4. `selected_files` / `config_fragments`
5. `context_hints`

### tie-breaker 规则
当多个候选 pack 同级竞争时，依次用以下规则裁决：
1. 谁的 `entry_conditions` 满足得更直接
2. 谁的 `anti_triggers` 更少触发
3. 谁的 `required_with` 能构成更小的 bundle
4. 谁更接近固定主任务
5. 若仍不清楚，则退回 L0，不强行深路由

### 上下文不足退回规则
当出现以下任一情况时，只允许 route 到 L0 或返回 partial：
- task 过于泛化
- 没有 selected_files / fragments
- hints 只指向领域，不能指向子问题
- 候选 pack 之间无法稳定区分

### 路由解释性要求
每个 route 结果必须写明：
- 为什么进入这个主 pack
- 为什么挂这些 required packs
- 为什么没有进入另一个近邻 pack

## 四、补齐全局 schema 与 pack 局部 contracts 边界

### 全局层稳定对象
第一阶段全局层只稳定 4 个 schema 文件：
- activation-request.schema.json
- context-bundle.schema.json
- artifact.schema.json
- handoff-bundle.schema.json

### 协议层正式对象
以下两个对象必须正式定义，但第一阶段可以先以协议文档 + canonical JSON example 形式冻结：
- Activation Result
- Validation Plan

### 局部 contracts 允许做什么
pack 局部 contracts 可以：
- 细化 pack 自己的输入 payload
- 细化 pack 自己产出的 artifact body 结构
- 细化 handoff 附带的信息块

### 局部 contracts 禁止做什么
pack 局部 contracts 不能：
- 改写全局 activation result / artifact / handoff 顶层语义
- 为顶层输出重新命名
- 自己发明与全局 schema 平级但不兼容的新结果容器

### 建议约束
pack 局部扩展只允许放在：
- `input.payload`
- `artifact.body`
- `handoff.details`
这类“局部扩展槽位”中，禁止直接污染全局顶层字段。

## 五、Breaking Change 分级

### A 级：非破坏性文档变更
示例：
- 描述更清楚
- 增加例子
- 加注释
不会影响消费者。

### B 级：兼容性扩展
示例：
- 在不改变既有语义的前提下，增加可选字段
- 增加新的 recommended_validators
需要更新文档，但通常不破坏现有 pack。

### C 级：破坏性变更
示例：
- 修改必填字段
- 修改状态枚举
- 修改 artifact 类型语义
- 修改顶层对象命名
这类变更需要显式版本升级和迁移说明。

### D 级：冻结期禁止变更
在 Phase 1 闭环未跑通之前，以下对象原则上禁止改：
- artifact 类型表
- activation result 状态枚举
- frontmatter 必填字段集合
- package.yaml 必填字段集合

## 六、本文补齐的结果
完成本文后，后续里程碑可以直接假定：
- Activation Result 已经具备正式消费模型
- Validation Plan 已经具备正式生成模型
- route 冲突有裁决规则
- pack 局部 contracts 不会污染全局协议
- breaking change 有明确分级

## 非目标
本文不定义：
- 具体 blueprint 节点内容
- 具体 pack 目录内容
- 具体 validator 检查项
这些由后续里程碑文档负责。
