# 11 M0 上下文：系统定义与硬边界

## 作用域
本文只回答三个问题：
1. 这个系统到底是什么
2. 这个系统绝对不是什么
3. 后续所有里程碑不能突破哪些硬边界

## 固定定位
Agent Pack 是一个：
- 由外部 agent **显式激活**
- 面向宿主 monorepo 的
- 非侵入式增强能力包系统

它不是业务主仓，也不是模板仓，更不是自治型分析平台。

## 系统的三层核心
1. **Blueprint Content System**
   - 承载领域知识节点
2. **Blueprint Index & Query System**
   - 负责 route、选择与最小 bundle
3. **Agent Consumption System**
   - 接 activation request，产生 artifact / validation / handoff

## 硬边界

### 边界 1：不承载宿主业务代码
仓库中不应出现：
- 真实 popup / background / content 业务代码
- 真实应用入口
- 为了“演示更真实”而加入的 demo app

允许存在的是：
- fixtures/contexts
- sample fragments
- regression scenarios
- artifact templates

### 边界 2：没有 activation request 就不运行
禁止：
- 默认 host discovery
- 默认 repo indexing
- 默认 domain routing
- 默认 pack auto-run

### 边界 3：bounded context 是上限，不是建议
pack 只能消费：
- 被显式提供的文件
- 被显式提供的片段
- 轻量 hints
- 被显式允许的 operations

不能靠“也许更完整”自行扩权。

### 边界 4：pack 是最小执行单位
系统不是围绕“工具函数”运行，而是围绕 pack 运行。  
route 的目标是 **主 pack + 必要横线**，而不是任意文档片段拼装。

### 边界 5：输出闭环只有四类核心对象
后续实现必须围绕以下对象：
- activation request
- context bundle
- artifact
- handoff

补齐缺口后，还需要正式固定：
- activation result
- validation plan

## 第一阶段唯一成功标准
第一阶段成功，不看覆盖率，不看 pack 数量，不看工具数量，只看：
- 是否存在一条稳定、重复可跑的 activation -> route -> bundle -> artifact -> validation -> handoff/complete 闭环

## WXT 的固定定位
WXT 是第一条 **domain pack line**，不是：
- 浏览器插件模板
- 业务样板工程
- starter repo

WXT 的作用是验证：
- pack 模型
- route 模型
- bundle 模型
- artifact 模型
- validator 模型
- cross-cutting 挂接模型

## 后续里程碑必须维持的防腐判断
所有新增设计都要先过三个问题：
1. 会不会让 agent-pack 变成业务主仓
2. 会不会让系统脱离 activation request 自己运行
3. 会不会让新领域必须重写骨架而不是只换内容

任何一个问题答案为“会”，都必须停止。

## 非目标
本文不定义：
- frontmatter 具体字段
- package.yaml 具体字段
- routing tie-breaker
- validator 结果细节

这些由别的文档定义。本文只定义不可突破的系统边界。
