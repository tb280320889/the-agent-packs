# 01 通用开发方法论与里程碑解耦规则

## 文档类型
通用规则

## 作用域
适用于所有里程碑。  
它不定义某个具体 pack 或某个具体 schema，而是定义 **整套项目推进的方法论、解耦方式、冻结纪律与复用原则**。

## 本文解决什么
1. 用什么通用方法论推进开发。
2. 为什么必须用里程碑解耦，而不是全仓同时铺开。
3. 每个里程碑的输入、输出、冻结对象、禁止事项如何定义。
4. 这套方法如何复用到第二条、第三条领域线，而不重写 operating model。

## 非作用域
- 不给某个具体 pack 的业务细节
- 不给某个具体 validator 的规则明细
- 不给某个具体 schema 的字段级定义

---

## 一、总方法论：先固定壳，再证明一条闭环切片

### 1. 合同先于实现
先冻结：
- 术语
- 对象
- 协议
- schema
- route 裁决规则
- handoff 语义

再做实现。  
没有稳定 contract 的实现，一定会在第二个 pack 出现形状漂移。

### 2. 图谱优先于自由搜索
系统不是“让 agent 自由搜文档”。  
系统是：
- 用 Blueprint 定义节点
- 用 index/query 层把节点关系图结构化
- 用 MCP 把受控读取能力暴露出去
- 让 agent 只消费最小上下文包

### 3. 渐进式披露优先于一次性灌全文
默认只给：
- L0/L1 摘要
- 必带横线摘要
- 极少量执行态 L2

L3 只在 validator 失败、依赖缺失、边缘情况成立时展开。

### 4. 垂直切片优先于横向铺面
长期 operating model 可以覆盖很多产品线。  
但首期实现必须先拿一条 **边界清楚、artifact 明确、validator 好写** 的切片证明闭环。

### 5. 验证与交接优先于“看起来能跑”
只会生成产物，不足以说明系统成立。  
必须同时具备：
- 输出结构约束
- validator
- activation result
- handoff 表达
- regression 基线

### 6. 可复制骨架优先于一次性做全
首期做的不是“完成很多领域”，而是证明：
- 同一套协议
- 同一套 package 模板
- 同一套 compiler/query/bundle/validator 方法
可以复制到其他领域线。

## 二、里程碑解耦规则

### 1. M0 的职责
冻结公共语言与公共契约。  
M0 不做厚实现，不做完整 pack，不做 UI。

### 2. M1 的职责
把 Blueprint 和最薄 entry/query 跑起来。  
M1 不负责完整 pack，也不负责复杂 validator。

### 3. M2 的职责
完成首个完整 workflow package。  
M2 不扩很多领域线，不重构系统层。

### 4. M3 的职责
把验证与 activation 闭环跑通。  
M3 不发明新协议，只消费 M0~M2 已冻结对象。

### 5. M4 的职责
冻结 Phase 1，并定义第二领域线准入。  
M4 不再大改核心 contract。

---

## 三、每个里程碑的统一输入 / 输出模型

### 输入
每个里程碑只能消费三类输入：
1. 上游冻结对象
2. 本里程碑必须知道的上下文文档
3. 本里程碑自己的目标与验收条件

### 输出
每个里程碑必须至少输出：
1. 代码 / 配置 / 文档交付物
2. 冻结对象清单
3. handoff 摘要
4. 未解决风险，但要明确是否阻塞下游

### 禁止事项
- 擅自改上游冻结 contract
- 为了省事绕开最小上下文原则
- 以“后面再整理”为由在当前里程碑偷塞临时对象
- 因为想并行开发就跳过 route/bundle/validator 这些壳层

---

## 四、对 agent 的统一工作方式要求

### 1. 先判断作用域，再读文档
不是所有文档都该读。  
先看当前里程碑开发指导文档，再按其要求读取上下文。

### 2. 只读必须知道的事实
当前里程碑拿到的信息量应尽量小。  
“知道太多”会让实现重新发散。

### 3. 永远输出结构化结果
每一步都应尽量给出：
- 输入
- 判断
- 产出
- 冻结对象
- 风险
- 下一步依赖

### 4. 不在运行时发明协议
协议与 schema 只能回到对应文档中维护。  
实现层只能消费，不应边跑边改。

### 5. 共享上下文必须落地为公共对象
任何会影响下一位 agent 决策的关键信息，至少要进入以下之一：
- `bd issue`
- `Context Snapshot`
- `Handoff`

只存在于聊天窗口里的上下文，不算已共享上下文。

## 五、跨领域复用的方法骨架

这套方法复制到其他领域时，保持不变的只有下面这些壳层：

### 固定不变
- activation request
- routing protocol
- context bundle protocol
- artifact envelope
- validation plan
- validator result
- activation result
- handoff bundle
- blueprint L0/L1/L2/L3 结构
- compiler -> index -> query MCP -> bundle 的链路
- workflow package 模板
- breaking change 治理

### 可以换的
- domain
- subdomain
- capability
- pack 内部 rules / templates / examples
- domain validators
- artifacts 的具体内容

### 不允许换的原因
一旦壳层不稳定，第二条领域线不会是“复用”，而会变成“再造一套系统”。

---

## 六、当前仓库推进的总策略

### 长期目标
形成完整的 agent operating model 母版。

### 当前目标
只证明：
1. 系统层母版完整
2. 编译 / 路由 / bundle / pack / validator / activation 能闭环
3. 这条闭环可复制到第二条领域线

### 当前不追求
- 一次建完整多产品线
- 过早的控制台、控制平面、全量图谱可视化
- 复杂全文检索
- 宿主扫描与自治执行

---

## 七、完成标准：什么叫“方法论成立”
当且仅当以下条件同时成立，才算这套方法论成立：

1. 系统层母版没有丢
2. 首期实现切片足够小
3. 关键对象都有稳定 contract
4. agent 只拿最小上下文包
5. route / bundle / validator / activation / handoff 能闭环
6. 第二条领域线可以复用同一骨架，而不是推翻重来
