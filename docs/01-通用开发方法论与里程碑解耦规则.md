# 01 通用开发方法论与里程碑解耦规则

## 作用域
本文件适用于：
- Agent Pack 首期 WXT 实现
- 未来 Tauri / Axum / 其他领域线复制
- 任何需要“先验证骨架，再复制到其他领域”的 pack-based system

本文件不提供领域知识；它只提供 **可跨领域复用的开发方法论**。

## 目标
把开发过程从“同时做很多看起来相关的事”，收敛成一条可重复使用的里程碑方法。

## 方法论名称
**Contract-first / Milestone-decoupled / Bounded-context 开发法**

## 七条总原则

### 1. 先冻结对象，再扩实现
对象先行：
- 名称
- 协议
- schema
- frontmatter
- package.yaml
- artifact 类型
- validator 结果

在这些对象还漂移时，不应大规模写 pack。

### 2. 一切以 activation request 为起点
没有 activation request：
- 不运行领域 pack
- 不默认扫描宿主
- 不默认做 route
- 不默认建立知识索引

### 3. 一切以 bounded context 为边界
pack 只能消费被调用者显式允许的：
- selected_files
- config_fragments
- context_hints
- allowed_operations

不能把“也许有帮助”当成越界理由。

### 4. 每阶段只追一个主目标
Phase 1 不是做大覆盖，而是做一条可验证闭环。
每个里程碑都必须只有一个主目标，避免并行发散。

### 5. 先做最薄可运行物，再做完整样板
顺序固定：
- 最薄契约
- 最薄入口
- 最薄查询
- 首个完整 pack
- 首批 validator
- 闭环
- 扩展

### 6. handoff 是正常机制，不是失败补丁
当问题主轴已经转移到别的 pack，必须交接，而不是继续吞并邻域问题。

### 7. 未来新领域只替换内容，不重写骨架
真正可复用的方法论必须满足：
- 替换 domain 名
- 替换 blueprint nodes
- 替换 pack goals
- 替换 templates
- 替换 domain validators

但不重写：
- activation model
- bundle model
- artifact model
- handoff model
- validator 分层
- 里程碑编排

## 标准里程碑模型

### M0：规则冻结与契约补齐
目标：让系统“怎么表达自己”稳定下来。  
输出：规范、协议、缺口补齐、变更规则。

### M1：知识骨架与最薄入口
目标：让系统“怎么选主 pack、怎么带上下文”跑起来。  
输出：目录、frontmatter、首批节点、activation entry、query / bundle。

### M2：首个完整 Pack
目标：让抽象系统第一次落成真实实体。  
输出：wxt-manifest 完整 pack、pack 内文档、模板、局部 contracts。

### M3：Validators 与主任务闭环
目标：让输出变成可验证对象，而不是“像结果的文字”。  
输出：activation result、validation plan、validator result、测试矩阵、主任务闭环。

### M4：冻结与扩域准入
目标：确认 Phase 1 已稳定，可复用，可复制。  
输出：冻结策略、breaking change 纪律、第二领域线准入门槛。

## 里程碑解耦规则

### 规则 A：每个里程碑只接收上一里程碑的“固定产物”
不要继承上一里程碑的开放讨论，只继承：
- 文档
- schema
- 模板
- 约束
- 已知风险

### 规则 B：每个里程碑必须有自己的作用域
每个里程碑文档都必须明确：
- 本里程碑负责什么
- 不负责什么
- 依赖什么
- 不应该修改什么

### 规则 C：每个里程碑都必须有 handoff
handoff 至少包含：
- 冻结对象
- 下一里程碑输入
- 风险与未决项
- 禁止事项

同时建议保留一份 Context Snapshot，记录：
- 当前事实状态
- 当前输入输出
- 风险/阻塞
- 后续待决项

### 规则 D：禁止跨里程碑偷跑
示例：
- M0 未冻结时，不该开始大写 pack
- M1 未形成稳定 bundle 时，不该开始做完整 validator
- M2 未有完整 pack 时，不该谈多领域扩张
- M3 未闭环时，不该谈第二领域线

### 规则 E：共享上下文不能只存在于对话
任何会影响后续 agent 决策的关键信息，至少要进入以下之一：
- `bd issue`
- `Context Snapshot`
- `Handoff`

只存在于聊天窗口里的上下文，不算已共享上下文。

## 所有里程碑开发指导文档的统一模板
每份“开发指导文档”必须包含：
1. 作用域
2. 本里程碑唯一目标
3. 输入前提
4. 必读上下文文档
5. 交付物
6. 实施步骤
7. 不要做的事
8. 完成定义
9. 向下游 handoff 的内容

## 所有里程碑上下文文档的统一模板
每份“上下文文档”必须包含：
1. 作用域
2. 本文只回答哪些问题
3. 固定事实
4. 对象契约 / 示例
5. 边界与非目标
6. 与其他文档的接口关系

## 通用的完成定义
一个里程碑只有在以下四类信息都清楚时才算完成：
- **对象清楚**：字段、名字、目录、模板不漂移
- **边界清楚**：本阶段不负责什么被明确写出来
- **交付清楚**：下一阶段拿得到固定输入
- **验证清楚**：有检查点，不靠“感觉差不多”

## 通用的复用判定
某个设计只有在同时满足以下条件时，才能视为“可复用方法”：
- 不依赖 WXT 特有词汇
- 换一个领域仍成立
- 不需要额外放宽 activation 边界
- 不需要重写 schema / artifact / validator 分层

这就是整个文档包的底层方法论。
