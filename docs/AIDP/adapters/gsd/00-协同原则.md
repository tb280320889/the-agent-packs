# AIDP 与 GSD 协同原则

## 文档目标
本文件用于明确：`docs/AIDP/` 与 GSD 不是两套相互竞争的流程，而是职责分层的协同关系。

目标不是改变 GSD 的工作流，而是通过 `AGENTS.md` 的上下文注入能力，把 AIDP 作为后续所有 GSD agent 的业务语义真实源。

## 一句话原则

`GSD 管任务与执行流，AIDP 管项目语义与运行语义。`

## 职责边界

### GSD 负责什么
- 任务拆分、依赖、状态推进
- phase / roadmap / execute / verify 工作流
- 执行中的任务真相源
- 当前做什么、下一步做什么、谁阻塞了谁

### AIDP 负责什么
- 项目是什么
- 当前版本做什么、不做什么
- 关键业务规则与系统边界是什么
- 哪些是假设、哪些是事实、哪些待确认
- 关键决策、变更原因、验证语义与运行态上下文
- 项目如何作为增强包生产者系统服务消费侧 agent

## 真相源规则

### 任务真相源
- 以 GSD 任务记录为准
- 不在 AIDP 中私下替代 GSD 记录任务状态

### 业务语义真相源
- 以 `docs/AIDP/` 为准
- 新进入仓库的 agent 默认先读 AIDP，再进入 GSD 既有工作流

### 代码与行为真相源
- 以代码、测试、可执行验证结果为准
- AIDP 不能覆盖代码事实，只能解释与约束代码演进

## agent 进入规则

1. 先读取根 `AGENTS.md`
2. 通过根 `AGENTS.md` 注入 `docs/AIDP/` 语义
3. 再进入 GSD 的原有工作流、phase 规划与执行流程
4. 如 AIDP 未覆盖关键来源事实，再按需回读旧文档或代码事实

默认通过 GSD 进入的 agent，首先属于内部开发 / 执行角色，而不是消费侧 agent；因此它读取 AIDP 是为了理解生产系统语义，而不是直接代替消费侧上下文获取过程。

## 不改变 GSD 的约束
- 不改 GSD 的命令语义
- 不改 GSD 的任务状态机
- 不把 AIDP 强行写成另一套 task tracker
- 不要求 GSD 产物复制一份完整 AIDP 正文

## AIDP 对 GSD 的注入方式
- 通过 `AGENTS.md` 指定默认阅读顺序
- 通过 `core/` 提供项目语义、边界、规则
- 通过 `runtime/` 提供当前假设、决策、变更、phase context、验证语义
- 通过 `adapters/gsd/` 规定与 GSD 工件的映射原则

## 冲突处理

### GSD 任务状态 与 AIDP 文档不一致
- 先以 GSD 为任务状态真相
- 再回写 AIDP runtime 工件，解释当前阶段变化

### AIDP 语义 与旧改造文档不一致
- 以 AIDP 当前结论为默认消费版本
- 若需要追溯原因，再回读 `docs/改造计划v1/`

### AIDP 语义 与代码事实不一致
- 以代码事实为准
- 立即更新 AIDP，并在变更摘要中说明偏差来源

## 防冗余规则
- 任务状态不要重复写进 AIDP 正文
- 长期稳定规则放 `core/`
- 会变化的状态放 `runtime/`
- 历史设计追溯放旧目录，不再重复复制为新的长期正文

## 完成判定
只有当以下条件同时满足，才说明 AIDP 与 GSD 已形成正确协同：

1. GSD 工作流无需修改即可继续使用
2. 新 agent 仅靠 `AGENTS.md` + `docs/AIDP/` 即可理解项目语义
3. 旧文档不再是默认入口或隐式依赖
4. 任务、语义、代码三类真相源边界清楚
