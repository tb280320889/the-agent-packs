# GSD：增强开发与变更落地

## 目的
说明在已有 `local AIDP + GSD` 的项目中，如何把增强开发、缺陷修复和新里程碑升级继续落到 GSD 工作流，而不改变 GSD 本身。

## 基本原则
- 先更新 local AIDP，再进入 GSD 工作流
- `.planning/` 是 GSD 运行工件，不是业务语义真相源
- 对范围、规则、验收有影响的变更，先完成 local AIDP 对齐
- 对消费侧上下文交付契约有影响的变更，必须同时补充样例化验证口径或验证计划

## 最小更新路径

### Type A
- 更新相关 runtime 工件
- 必要时更新 `STATE.md` 或局部 phase 文档
- 再进入小范围 GSD 执行

### Type B
- 先生成增强开发任务启动单
- 更新 local AIDP 的相关 `core/` / `runtime/`
- 再更新 `.planning/` 中受影响的 phase / state / roadmap 工件

### Type C
- 先完成 local AIDP 版本升级与结构性对齐
- 更新 `PROJECT.md`、`REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md` 的对应语义
- 如属于新里程碑，再使用 GSD 的既有里程碑工作流

## 推荐顺序
1. 读取 `AGENTS.md`
2. 读取 local AIDP 当前基线
3. 对照 `protocol/10-增强开发与迭代协议.md` 判断变更类型
4. 更新 local AIDP
5. 再更新 `.planning/` 与 GSD 工作流工件
6. 最后进入具体实现与验证

## 消费契约落地要求
- 若本轮变更影响主域路由、context bundle 裁剪、MCP 输出或 capability 附挂规则，除了更新 `core/` 文档，还应补充至少一个样板域验证口径
- 当前默认样板域是 `WXT -> wxt-manifest`
- 后续实现任务进入 GSD 前，应能回答：本轮改动会让消费侧多拿什么、少拿什么、为什么这样拿
