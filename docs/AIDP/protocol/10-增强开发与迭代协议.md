# 增强开发与迭代协议

## 目的
定义本项目在已经拥有 local AIDP 之后，如何在不破坏既有语义基线的前提下继续做：
- 小修复
- 新增能力
- 新里程碑推进
- 结构性升级

## 核心思想
- 先更新 local AIDP，再继续 GSD 工作流
- 先读现状，再决定改什么
- 先判断变更类型，再决定更新范围
- 任何影响范围、规则、验收或里程碑的变更，都必须回写运行态工件

## 变更类型

### Type A：小修复 / 小增强
- 不改变项目主闭环
- 不改变关键规则与边界
- 主要更新局部实现、局部说明、局部验证

### Type B：阶段增强
- 新增一个明确能力面
- 影响现有一个或多个子流程
- 需要更新 phase context、验收和局部 roadmap 语义
- 影响消费侧上下文交付契约但不重写整体身份模型

### Type C：结构性变更 / 里程碑升级
- 改变范围边界、关键规则、主流程或治理方式
- 需要重新审视 `core/`、`runtime/` 以及 GSD 协同方式
- 必要时提升 local AIDP 版本
- 若影响生产者 / 消费侧 / 开发者三类身份模型或 AIDP 在系统中的顶层定位，默认至少按 Type C 处理

## 标准流程
1. 读取当前 local AIDP 基线
2. 读取当前 GSD 状态与相关 `.planning/` 工件
3. 判断本轮变更类型
4. 生成增强开发任务启动单
5. 仅更新受影响的 local AIDP 文档
6. 回写运行态工件
7. 再继续 GSD 的原有规划或执行流程

## 最低读取要求
至少读取：
- `core/01-项目总览.md`
- `core/05-范围与边界.md`
- `core/06-业务规则与关键对象.md`
- `core/08-技术约束与工程约定.md`
- `core/10-验收标准.md`
- `core/12-默认假设与待确认问题.md`
- `runtime/02-决策日志.md`
- `runtime/03-变更摘要.md`

## 输出要求
本协议下至少应输出：
- 当前基线摘要
- 本轮变更类型
- 影响分析
- 需要更新的文档清单
- 新的默认假设与待确认问题
- 变更摘要
- 对 GSD 下一步的建议动作

## 禁止事项
- 不读现状就重写 local AIDP
- 让 GSD 先变更而 local AIDP 滞后
- 用新的默认假设覆盖旧确认事实
- 因小改动就重写整套文档集
