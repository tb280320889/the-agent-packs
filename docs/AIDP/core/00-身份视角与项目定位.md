# 身份视角与项目定位

## 项目顶层定位
`the-agent-packs` 的第一身份不是泛化的 agent 编排系统，而是 `agent-packs` 增强包的生产者系统。

项目的核心职责，是把经过治理、分域、分层、可验证的知识与 workflow 能力，生产为可被下游 agents 通过 MCP 等方式按需消费的增强 pack。

路由、索引、注册表、bundle、activation、validator、handoff 等机制，属于这个生产系统的内部实现手段，而不是项目最终产品定义本身。

## 第一性目标
- 生产可被消费侧 agent 精准调用的增强 pack
- 让消费侧 agent 仅获得当前任务所需、当前主域相关、当前 capability 相关的最小完整上下文
- 保证上下文交付满足“不冗余、不遗漏、不串域”
- 让开发本项目的开发 agent 清楚自己是在建设生产系统，而不是直接扮演消费侧 agent

## 三类核心身份

### 1. 生产者角色
生产者角色是本项目自身及其治理体系。

它负责：
- 吸收和治理知识资产
- 定义主域、横线能力与 package 边界
- 维护注册表、路由约束、上下文裁剪规则和验证机制
- 生产可暴露、可消费、可验证的增强 pack

生产者角色的成功标准，不是“内部系统看起来完整”，而是“外部消费侧能正确拿到目标增强上下文”。

### 2. 消费侧 agent 角色
消费侧 agent 是本项目产物的使用者。

它通过 MCP 或其他受控调用方式，按任务目标消费某个主域或 capability 对应的增强上下文，而不是默认阅读整个仓库或整个 AIDP 文档集。

消费侧 agent 的成功标准，是在不理解本仓库全部内部治理细节的前提下，仍能获得完成目标任务所需的最小完整上下文。

### 3. 开发本项目的开发 agent 角色
开发本项目的开发 agent 是本仓库的内部建设者。

它需要理解：
- 本项目在生产什么
- 为什么要这样分域、分层与裁剪上下文
- 哪些是稳定规则、哪些是默认假设、哪些仍待确认

它的职责是建设和维护生产系统本身，而不是把自己默认代入最终消费侧 agent 的视角。

## 身份边界
- 生产者角色不等于消费侧角色
- 开发 agent 不等于消费侧 agent
- AIDP 是生产侧语义真相源，不是消费侧默认全文入口
- 消费侧真正应消费的是经治理和裁剪后的 pack / context bundle，而不是整个 `docs/AIDP/`

## 双闭环模型

### 内部生产闭环
知识吸收 -> 治理归档 -> Blueprint / Registry 建模 -> 编译与索引 -> 路由与上下文裁剪 -> activation / validator -> 增强 pack 暴露与交付语义

### 外部消费闭环
消费请求 -> 识别主域与 capability -> 获取目标 pack / context bundle -> 执行任务 -> 返回结果 / handoff

## 渐进披露的正式含义
在本项目中，渐进披露不是抽象口号，而是生产侧对消费侧的正式交付契约。

它要求：
- 默认不暴露全仓语义
- 只按任务、主域、能力和阶段交付所需知识
- 上下文可以扩展，但必须按需扩展，不做一次性全量灌入

## AIDP 的角色
`docs/AIDP/` 的首要职责，是作为生产系统的项目语义真实源，帮助内部维护者与开发 agent 正确理解本项目。

同时，AIDP 还必须清楚定义：最终产物如何被消费侧 agent 正确消费，消费边界在哪里，何种情况下才允许继续扩展上下文可见面。

## 当前整改要求
后续所有文档整改，都必须满足以下判断：
- 是否明确项目首先是增强包生产者系统
- 是否区分开发 agent 与消费侧 agent
- 是否把渐进披露写成正式交付契约
- 是否支持“最精简、不冗余、不遗漏”的上下文交付目标
