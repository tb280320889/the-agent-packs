# 关键流程与状态流

## 主流程
1. 编译 Blueprint
2. 构建 SQLite 索引
3. 第一轮全局 route，确认主域 / 主域 orchestrator
4. 第二轮域内 route，确认主 workflow package
5. 第三轮 capability 挂接，决定 required / may_include
6. 构建 Context Bundle
7. 执行 Activation
8. 执行 Validator
9. 输出 Activation Result / Handoff

## 文档与协作流程
1. 读取 AIDP 入口
2. 判断当前任务类型
3. 从 GSD 获取任务真相
4. 读取最小必要代码事实与旧文档来源
5. 产出或修改实现/文档
6. 更新运行态工件
7. 写入验证与交接结果

## 状态分类
- `confirmed`：已确认事实
- `assumed`：当前采用但未确认的默认假设
- `open`：待确认问题
- `blocked`：不确认则无法继续
- `completed`：阶段目标已完成

## 状态流要求
- 从 `assumed` 转 `confirmed` 时要更新假设账本
- 结论变化时要追加决策日志与变更摘要
- 阶段完成时要更新 phase context 与验证记录

## 迁移与回滚要求
- 迁移顺序默认遵循：角色与边界 -> 节点分类与 route 分层 -> 注册表与命名治理 -> 资料吸收治理 -> 实现迁移与准入演练
- 回滚必须尽量按层进行，而不是整体撤回；至少区分文档层、注册表层、路由层、资产层
- capability 若重新进入第一轮主竞争，属于必须回退的信号
