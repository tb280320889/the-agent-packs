# 角色、身份与场景

## 身份总览

本项目至少存在三类不能混淆的身份：
- 生产者角色：生产和治理增强 pack 的系统与维护体系
- 消费侧 agent 角色：消费增强 pack 的下游 agents
- 开发本项目的开发 agent 角色：建设和维护生产系统本身的 agents

下文所有参与者，都必须放回这三类身份中理解，不能只看名字不看职责边界。

## 主要参与者

### 1. 项目维护者
负责定义目标、审核边界、确认关键决策，并判断何时允许进入下一阶段。

### 2. 项目内部维护 agent
负责读取 AIDP、判断请求类型、选择最小阅读集、维护共享上下文与运行态工件。

它属于开发本项目的开发 agent 角色。

不负责把自己变成外部世界资料采集器，也不应直接替代消费侧 agent。

### 3. 迭代开发 agent
负责实现路由、注册表、validator、文档结构与迁移相关改动，并回写验证与交接信息。

它属于开发本项目的开发 agent 角色。

不应越过冻结边界直接改顶层协议或状态语义。

### 4. 文档吸收 agent
负责接收 skill、MCP、llm.txt、最佳实践等外部材料，经治理管道后转化为项目正式资产。

它属于生产者治理体系的一部分。

不替代实现层 agent，也不负责绕过注册表和命名空间规则直接新增 package。

### 5. 用户侧外部 agent
代表用户通过 MCP 等方式消费本项目产出的增强 pack 或上下文切片。

它不是仓库内部正式治理角色，不直接冻结本项目内部规则，也不应默认暴露整个 `docs/AIDP/`。

## 各身份的输入、输出与可见范围

### 生产者角色
- 主要输入：领域知识、外部资料候选、注册规则、验证规则、运行态反馈
- 主要输出：增强 pack、上下文裁剪规则、验证机制、受控暴露面
- 默认可见范围：项目全局语义、边界、运行态、治理规则

### 消费侧 agent 角色
- 主要输入：任务目标、主域线索、capability 需求
- 主要输出：任务结果、activation result、handoff
- 默认可见范围：目标域或目标能力对应的增强上下文，不默认可见全仓治理语义

### 开发本项目的开发 agent 角色
- 主要输入：AIDP、GSD 工件、代码、测试、运行态工件
- 主要输出：实现改动、文档改动、验证记录、运行态回写
- 默认可见范围：生产系统全局语义与实现事实

## 典型混淆案例
- 把开发 agent 当成消费侧 agent，只从某个领域包视角理解全局项目
- 把消费侧 agent 当成内部维护者，默认暴露完整 AIDP 和治理工件
- 把“最小上下文”误写成“开发者也只需要知道局部 pack”，导致后续实现失去全局语义支撑

## 典型场景

### 场景 A：新 agent 进入仓库
目标是在最短时间建立正确系统模型，先判断自己属于哪类身份，再理解当前阶段、关键约束、默认假设和运行态状态。

### 场景 B：继续推进改造计划
目标是把旧设计结论与当前代码实现差异收敛为可执行任务，并按 GSD + AIDP 方式推进，同时不混淆生产侧语义与消费侧语义。

### 场景 C：新增领域线或 capability
目标是在不破坏现有顶层协议和路由边界的前提下，安全接入新的主域或能力线，并保证未来消费侧只获得相关上下文切片。

### 场景 D：吸收外部资料
目标是把外部知识转换为可被 route、bundle、validator 或项目文档消费的受控资产。

### 场景 E：验证与交接
目标是判断某个阶段是否完成、是否可回退、是否可进入下一阶段，并留下对下一个 agent 可执行的交接信息。
