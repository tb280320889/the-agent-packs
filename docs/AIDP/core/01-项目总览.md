# 项目总览

## 项目名称
`the-agent-packs`

## 一句话产品描述
一个 `agent-packs` 增强包生产者系统：基于 Blueprint、SQLite 索引、workflow package 与 validator，将经治理的领域知识与能力生产为可被消费侧 agents 按渐进披露模式精准消费的增强 pack。

## 项目类型与当前阶段
- 项目类型：`brownfield` + `milestone-extension`
- 当前阶段：Agent Pack 改造计划 v1 的文档协议统一阶段，AIDP 已成为新的默认文档入口

## 北极星目标
让仓库能够在不破坏现有 activation 顶层协议与最小上下文原则的前提下，稳定接入更多主域、更多能力线和更多 workflow package，并把这些能力组织成可被消费侧 agents 精准调用的增强 pack，同时支持后续开发 agent 持续开发、验证、审计与交接。

## 项目要解决的问题
当前仓库已经具备从 Blueprint 到 activation result 的最薄闭环，但在多域扩展、命名治理、知识吸收、消费侧上下文裁剪、运行态文档维护和 agent 接力协议层面仍不够稳定。

当前最关键的问题不是“有没有系统”，而是“这个系统是否能稳定生产出可被消费侧 agents 正确消费的增强 pack，并让开发本项目的开发 agents 不混淆自己的角色”。

## 三类核心身份与核心任务
- 生产者角色：负责治理知识、组织 package、定义边界、维护验证，并生产可被消费的增强 pack
- 消费侧 agent 角色：负责通过 MCP 等方式消费目标主域或 capability 对应的最小完整上下文
- 开发本项目的开发 agent 角色：负责建设和维护增强包生产系统本身，不应把自己默认代入消费侧视角

## 当前最小可用闭环
`Blueprint Markdown -> SQLite index -> route query -> context bundle -> workflow package activation -> validator -> activation result / handoff`

这条闭环描述的是内部生产与交付主链路；消费侧 agent 实际消费的对象，是这条链路产出的目标域增强 pack / context bundle，而不是仓库全部语义。

## 指定技术栈
- 语言：Go 1.25
- 数据与索引：Markdown + frontmatter + SQLite
- 包治理：`workflow-packages/registry.json` + `package.yaml`
- 测试：Go `testing`

## 当前阶段目标
把旧的 `docs/改造计划v1/` 与 `.planning/codebase/` 中的事实，实例化为本项目专属 AIDP 文档集，并用 AIDP 明确本项目作为增强包生产者系统的身份、消费契约和开发边界，让新的 agent 入口、项目语义入口和 GSD 映射入口统一指向 `docs/AIDP/`。

## 关键约束
- 不推翻现有 activation 顶层协议
- 不放宽 bounded-context / activation-first 原则
- 不允许 capability 在第一轮全局路由中与主域直接竞争
- GSD 仍是任务真相源
- AIDP 是生产侧语义真相源，不等于消费侧全文入口
