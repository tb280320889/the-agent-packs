# 范围与边界

## Must
- 统一以 `docs/AIDP/` 作为新的项目文档协议入口
- 保留现有 activation 主链路，不推翻顶层协议
- 明确 `global -> domain -> capability` 分层候选模型
- 明确 capability attach-only 语义
- 把注册表定义为 package 身份与映射真相源
- 为多-agent 接力建立运行态工件维护规则
- 给 GSD 提供清晰映射入口
- 明确生产者角色、消费侧 agent、开发 agent 三类身份边界
- 明确消费侧默认只获得目标域和目标任务所需的上下文切片

## Later
- 第二领域线正式准入与收口
- 动态 validator 注册机制
- 更通用的 domain 推断与配置化策略
- Blueprint 索引版本迁移机制
- 更完善的外部资料资产治理工具化

## Out of Scope
- 彻底重写现有 Go 运行时
- 放开 agent 直接自由搜索全部 Blueprint 文档作为主路由方式
- 在当前版本中承诺所有未来主域同时落地
- 把 capability 提升为全局主入口竞争者
- 让消费侧 agent 直接自由阅读完整 `docs/AIDP/` 作为默认路径
- 用“更多上下文更保险”替代渐进披露与最小完整交付

## 角色边界

### 消费侧边界
- 消费侧默认不暴露完整仓库治理语义
- 消费侧只消费目标主域、目标能力、目标任务所需的上下文

### 开发侧边界
- 开发本项目的开发 agent 必须理解生产系统全局语义
- 开发 agent 不应把自己默认缩减为某个单域消费者视角

### 生产与消费边界
- AIDP 属于生产侧语义层，不等于消费侧默认交付面
- 对外交付的应是增强 pack / context bundle，而不是内部治理正文

## 边界规则
- 只能在冻结对象允许的范围内演进
- 范围变化必须进入变更摘要与决策日志
- 涉及新领域线或 breaking 讨论时，必须显式说明对当前闭环的影响
