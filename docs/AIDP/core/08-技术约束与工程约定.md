# 技术约束与工程约定

## 技术栈约束
- 主语言：Go 1.25
- 核心存储：SQLite
- 知识源：Markdown + frontmatter
- package 清单：YAML + JSON

## 工程结构约定
- CLI/MCP 入口在 `cmd/agent-pack-mcp/main.go`
- 核心模块在 `internal/activation/`、`internal/compiler/`、`internal/query/`、`internal/registry/`、`internal/validator/`
- Blueprint 资产在 `blueprint/`
- workflow packages 在 `workflow-packages/`
- Brownfield 扫描结果在 `.planning/codebase/`

## 开发约定
- Go 代码遵循 `gofmt`
- 错误处理以结构体 + `error` 返回为主
- 测试使用 Go `testing`
- 非交互式命令优先

## 文档约定
- 项目语义、边界、规则、运行态维护统一以 `docs/AIDP/` 为准
- 旧的 `docs/改造计划v1/` 作为历史来源保留
- 关键变化必须回写运行态工件
- 外部资料吸收必须保留 `source_uri -> target_scope -> target_ref -> 落位` 的完整链路

## 消费暴露约定
- 对消费侧暴露的知识默认不是完整 AIDP，而是目标域增强 pack / context bundle
- MCP 或其他暴露层输出的上下文，应保留来源、裁剪依据和附挂 capability 的可追踪信息
- 若上下文引入横线 capability，必须能解释其与当前主域任务的关系
- 不允许因实现便利，把无关主域知识直接混入消费侧输出

## 当前技术限制
- frontmatter / YAML 解析仍较手工、较脆弱
- SQLite 索引重建缺少更强事务保护
- domain 推断与 validator 注册仍有硬编码倾向
- 缺少索引 schema 版本迁移机制

## 注册表与命名工程约定
- package 目录名、`package.yaml` 中的 `name`、注册表 `name` 必须一致
- `canonical name` 是 package 身份真相源
- `category + visibility_scope + activation_mode` 必须足以判断 package 的候选空间与主竞争资格
- `reserved = true` 的对象不能按普通 workflow package 直接放行
