# 风险与依赖

## 主要技术风险
- `package.yaml` 解析较手工，字段扩展易脆弱
- frontmatter 解析简化，复杂语法支持不足
- SQLite 索引重建流程缺少更强事务保护
- 默认主包回退曾有硬编码倾向，容易误导非 WXT 场景

## 路由与扩展风险
- 主域推断逻辑当前偏 WXT 特化
- validator 注册表静态写死，扩展成本高
- 大规模节点关系校验能力不足
- 若消费契约只停留在文档层而未继续映射到实现和验证层，后续扩域时容易再次回退为“多给一点上下文更保险”的策略

## 安全与稳健性风险
- 外部输入路径读取缺少更严格边界
- MCP JSON 输入缺少体积限制与更强校验

## 关键依赖
- Go 1.25
- `modernc.org/sqlite`
- `workflow-packages/registry.json`
- `blueprint/` 中的 frontmatter 质量

## 文档依赖
- 历史改造设计来源：`docs/改造计划v1/`
- 代码事实来源：`.planning/codebase/`

## 当前文档治理风险
- AIDP 虽已成为默认入口，但若长期不把旧目录中的有效规则继续迁入，旧目录仍会形成隐性依赖
- 若 runtime 工件不持续维护，AIDP 会再次退化为静态文档包
- 若未持续维护 `WXT` 等样板域的消费契约样例，后续 agent 很难判断“最小但完整”在具体场景下如何成立

## 消费契约落地风险
- MCP 输出层若不保留来源和裁剪依据，消费侧出现上下文冗余或遗漏时将难以审计
- 若未定义样板域的应交付 / 不应交付边界，开发 agent 可能重新混淆内部治理正文与消费侧输出

## 扩域治理风险
- 当前已具备第二领域线准入机制，但仓库内仍未落地真实第二主域样板；若误把“机制 ready”表述成“样板已完成”，会造成范围与验收失真
- 后续新增 package、capability 或主域时，若不复用既有准入检查表、注册表校验和验证节奏，会重新引入命名漂移与越界路由风险
