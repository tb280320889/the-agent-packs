# 验收标准

## 项目层验收
- AIDP 已成为新的默认项目入口
- 新 agent 能在不依赖旧对话的情况下理解项目定位、边界、规则与当前状态
- GSD 任务真相源与 AIDP 语义层不冲突
- 项目定位已明确为增强包生产者系统，而不是泛化编排系统
- 角色边界已明确区分生产者、消费侧 agent 与开发 agent

## 当前阶段验收
- 已生成项目专属 `docs/AIDP/` 文档集
- 根目录 `AGENTS.md` 已把默认索引切换到 `docs/AIDP/`
- 旧 `docs/改造计划v1/` 已从默认入口降为历史来源
- 已具备最少运行态工件：假设账本、决策日志、变更摘要、phase context、验证记录
- 已明确 AIDP 与 GSD 的协同边界，不改变 GSD 工作流
- 已具备 local AIDP 的版本治理与增强开发机制
- 已明确 AIDP 是生产侧语义真相源，不等于消费侧默认全文入口
- 已明确消费侧上下文交付遵循渐进披露、最小且完整的契约

## Requirement 级验收

### REQ-01 文档入口统一
- 验收方式：检查 `AGENTS.md`、`docs/AIDP/README.md`、`docs/AIDP/AGENTS.md`

### REQ-02 关键语义完整
- 验收方式：检查 `core/01` 到 `core/12` 是否覆盖项目定义、范围、规则、约束、验收、风险、假设

### REQ-03 运行态可维护
- 验收方式：检查 `runtime/` 是否提供可持续更新的工件

### REQ-04 GSD 可映射
- 验收方式：检查 `adapters/gsd/` 是否说明 `PROJECT.md`、`REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md` 映射

### REQ-05 可安全退役旧文档
- 验收方式：检查 `runtime/07-旧文档退役检查表.md` 中关键项是否通过

### REQ-06 扩域与复用应处于受护栏状态
- 验收方式：检查当前结论是否明确区分“准入机制 ready”与“真实第二主域样板已落地”，避免虚构完成状态

### REQ-07 角色不混淆
- 验收方式：检查 `core/00`、`core/01`、`core/03` 是否清楚区分生产者、消费侧 agent、开发 agent 三类身份

### REQ-08 消费侧只获目标域上下文
- 验收方式：检查 `core/04`、`core/05`、`core/06` 是否明确消费侧默认只获得目标域和任务所需上下文

### REQ-09 内部治理语义不默认外泄
- 验收方式：检查 `README.md`、`core/06`、`core/08` 是否明确 AIDP 不等于消费侧默认交付面

### REQ-10 上下文最小且关键不遗漏
- 验收方式：检查 `core/06` 与 `core/10` 是否同时要求最小性与完整性，而不是只强调减少上下文

### REQ-11 WXT 样板消费契约可示例验证
- 验收方式：检查 `runtime/10-WXT-消费契约验证样例.md`，确认文档已给出 `WXT -> wxt-manifest` 的消费契约样例，且能区分“应交付内容”和“不应默认交付内容”

## 快速验证方式
- 文档检查：AIDP 目录结构存在且文件可读
- 仓库验证：`go test ./...`
- 语义检查：以 `wxt` 为例，检查文档是否支持“消费侧仅获得 WXT 相关关键上下文，而非无关域与完整治理正文”
- 样例检查：阅读 `runtime/10-WXT-消费契约验证样例.md`，确认已形成最小但完整、可解释、可追踪的文档级验证口径

## 不覆盖项
- 不在本阶段承诺所有未来领域线已落地
- 不在本阶段承诺所有技术债已修复
- 不在本阶段替代具体实现迁移任务本身

## 当前基线判定
- 当前本地基线可视为：`ready-with-guardrails`
- 解释：核心骨架、实现约束、验证机制与协同规则已形成可复用基线，但当前仍处于“增强包生产者系统语义校准与护栏明确”阶段；真实第二主域样板及更大范围扩域仍属于后续常规开发，而不是本阶段已完成事实
