# 语义整改计划

## 目标
本轮整改的目标，不是零散修正文案，而是把 `docs/AIDP/` 的顶层语义从“泛化编排系统叙事”校准为“增强包生产者系统叙事”，并明确生产者、消费侧 agent、开发 agent 三类身份边界。

## 分批计划

### Batch 1 顶层语义校准
- `README.md`
- `AIDP-MANIFEST.yaml`
- `core/00-身份视角与项目定位.md`
- `core/01-项目总览.md`
- `core/02-北极星与成功定义.md`
- `core/03-用户与场景.md`

完成标准：
- 项目顶层定位统一为增强包生产者系统
- 三类身份边界可从顶层文档直接读出

### Batch 2 规则与闭环校准
- `core/04-产品定义与最小闭环.md`
- `core/05-范围与边界.md`
- `core/06-业务规则与关键对象.md`
- `core/08-技术约束与工程约定.md`
- `core/10-验收标准.md`
- `core/12-默认假设与待确认问题.md`

完成标准：
- 双闭环模型、上下文交付契约与角色边界进入核心规则层
- 验收标准可检验消费正确性与身份不混淆

### Batch 3 协同与运行态收口
- `AGENTS.md`
- `protocol/01-输入输出定义.md`
- `protocol/05-输出生成协议.md`
- `protocol/10-增强开发与迭代协议.md`
- `adapters/gsd/00-协同原则.md`
- `adapters/gsd/01-全局产物映射.md`
- `runtime/01-默认假设账本.md`
- `runtime/02-决策日志.md`
- `runtime/03-变更摘要.md`
- `runtime/04-phase-context.md`
- `runtime/06-验证记录.md`

完成标准：
- 后续 agent 能根据运行态文档理解本轮语义整改
- GSD 协同与运行态记录不会回退到旧叙事

## 本轮默认执行顺序
1. 先改 Batch 1
2. 再改 Batch 2
3. 最后改 Batch 3
4. 完成后做一次全文一致性检查

## 风险控制点
- 不把 AIDP 误写成消费侧全文入口
- 不把“更多上下文”误当成“更安全的交付”
- 不把开发 agent 误写成消费 agent
- 不只改顶层措辞而不改规则与验收
