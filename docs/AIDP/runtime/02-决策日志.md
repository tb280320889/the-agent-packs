# 决策日志

## D-01 采用项目专属 AIDP 作为新的默认文档入口
- 日期：2026-03-16
- 状态：accepted
- 背景：旧 `docs/改造计划v1/` 已能表达改造设计，但不满足 AIDP 所要求的项目语义层、运行态工件层和适配层统一入口。
- 决策：新增 `docs/AIDP/` 作为项目专属 AIDP 文档集；后续 agent 默认从该目录进入。
- 影响：`AGENTS.md`、文档索引、运行态维护方式、GSD 映射入口均需要对齐。

## D-02 保留 `docs/改造计划v1/` 作为历史来源，不再作为默认主入口
- 日期：2026-03-16
- 状态：accepted
- 背景：旧文档仍包含重要阶段结论与来源事实，直接删除会丢失上下文。
- 决策：保留旧目录，但将其角色调整为历史来源与追溯材料。
- 影响：若 AIDP 与旧文档有差异，以 AIDP 当前结论和 runtime 工件为准。

## D-03 项目顶层定位校准为增强包生产者系统
- 日期：2026-03-16
- 状态：accepted
- 背景：当前 AIDP 对项目的叙事偏向“编排系统”和“内部开发入口”，不足以表达本项目真实核心目标。
- 决策：将 `the-agent-packs` 的顶层定位明确为 `agent-packs` 增强包生产者系统；编排、路由、bundle、activation、validator 等均视为内部实现机制。
- 影响：README、MANIFEST、core 关键文档、验收标准与运行态文档都需要同步改写。

## D-04 明确区分生产者、消费侧 agent、开发 agent 三类身份
- 日期：2026-03-16
- 状态：accepted
- 背景：若角色边界不清晰，后续开发容易把内部治理层、消费层和开发层混淆。
- 决策：在 local AIDP 中显式区分生产者角色、消费侧 agent 角色、开发本项目的开发 agent 角色，并将其作为后续文档整改的基础。
- 影响：`core/00`、`core/03`、`core/05`、`core/06`、`core/10` 等文档需以此为准。

## D-05 将渐进披露提升为正式上下文交付契约
- 日期：2026-03-16
- 状态：accepted
- 背景：仅将最小上下文视为内部原则，无法充分约束消费侧实际获得的上下文质量。
- 决策：将渐进披露和“最小但完整”定义为生产侧对消费侧的正式交付契约。
- 影响：后续规则、验收和技术约定都必须支持“最精简、不冗余、不遗漏”的交付目标。
