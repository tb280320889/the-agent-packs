# 52 M4 上下文：第二领域线准入与可复用骨架

## 文档类型
单里程碑所需上下文文档

## 作用域
定义 Phase 1 之后，如何把这套方法复制到第二条领域线。  
本文件只定义“如何复制骨架”，不决定第二条领域线一定是谁。

## 本文必须回答的问题
1. 第二领域线该怎么选。
2. 什么条件下允许准入。
3. 接入时复用哪些东西，不复用哪些东西。
4. 第二领域线接入步骤是什么。
5. 如何判断这套方法真的可复用。

---

## 一、第二领域线的选择原则

优先选择满足以下条件的领域：
- 边界清楚
- 输入形状明确
- artifact 好定义
- validator 好写
- 易接横线
- 不要求立即做复杂宿主能力

### 不优先选择
- 依赖过多外部系统
- route 模糊
- artifact 难以稳定
- 必须先做大量宿主运行时能力的领域

### 候选领域评分建议（用于准入前筛选）
可按 1~5 分做快速评分，总分越高越优先：
- 边界清晰度
- 输入 shape 可定义性
- artifact 可验证性
- validator 可实现性
- 横线挂接难度（安全/发布/身份）
- 对宿主运行时深依赖程度（反向计分）

### 候选领域评审输出模板
准入评审至少应留下以下记录，避免只在对话里口头决定：

| 字段 | 说明 |
|---|---|
| 候选领域 | 例如 `security-permissions`、`release-store-review` |
| 候选类型 | 主域 workflow / 横线 capability / orchestrator |
| 对应 Blueprint 节点 | 至少列出 L0/L1 入口节点 |
| 建议首个 package 名 | 必须符合注册表与命名规则 |
| 主 artifact | 首个可验证产物 |
| 建议 validator | 至少一个 domain validator |
| 固定主任务 | 进入 regression 的最小可重复场景 |
| 总分 | 6 项评分汇总 |
| 准入结论 | 通过 / 暂缓 / 拒绝 |
| 回退触发条件 | 一旦触发即停止实施并回到 breaking 或草案状态 |

### 当前仓库的优先筛选提示
若目标是先验证“骨架可复制”，优先选择同时满足以下事实的候选：
- Blueprint 已存在 L0/L1 入口节点。
- 已被首个 pack 作为 required 或 handoff 相邻线引用。
- 主 artifact 可以在不新增宿主运行时能力的前提下定义。
- validator 可以复用现有 validation envelope，只新增领域规则。

这类候选通常比全新主域更适合作为第二领域线准入演练对象，因为它更容易证明“复用骨架”而不是“顺手重构系统层”。

## 二、第二领域线准入门槛

只有同时满足以下条件，才允许进入：

1. M0~M4 冻结对象已稳定
2. 固定主任务闭环与 regression 已存在
3. workflow package 标准模板已可直接复制
4. Blueprint compiler / query / bundle 壳层可直接复用
5. 新领域只需要新增 domain-specific 内容，不需要改系统层

### 准入检查清单（必须全部为“是”）
| 检查项 | 结论（是/否） | 证据 |
|---|---|---|
| M0~M4 冻结对象可引用且无冲突 |  |  |
| 固定主任务回归可稳定通过 |  |  |
| workflow package 模板可直接复制 |  |  |
| route/query/bundle 壳层可复用 |  |  |
| 新领域无需改系统层与冻结壳层 |  |  |
| 可定义至少一个固定主任务用于回归 |  |  |

## 三、接入时必须复用的骨架

### 必须原样复用
- Activation Request envelope
- Routing Result envelope
- Context Bundle envelope
- Validation Plan envelope
- Validator Result envelope
- Activation Result envelope
- Handoff Bundle envelope
- Blueprint L0/L1/L2/L3 分层
- Blueprint Query MCP surface
- workflow package 标准模板
- tests 目录基本形状

### 复用边界解释
- “必须原样复用”表示：名称、顶层语义、最小结构不能改。
- “允许局部替换”表示：仅替换 domain 内容，不影响全局 envelope 与查询壳层。

### 允许局部替换
- domain
- subdomain
- capability
- domain triggers / anti_triggers
- domain-specific artifacts
- domain validators
- domain examples / fixtures

## 四、第二领域线接入步骤

1. 选定主域与首个 pack
2. 建最小 Blueprint 节点集
3. 复用 package 模板建包
4. 写 domain contracts
5. 接入现有 route / bundle / activation
6. 写 domain validator
7. 选一个固定主任务
8. 加入 regression

### 每一步的输入/输出定义
1. 选定主域与首个 pack
   - 输入：候选领域评分、准入检查清单
   - 输出：候选领域决策记录
2. 建最小 Blueprint 节点集
   - 输入：L0/L1 模板、主任务描述
   - 输出：最小节点与关系定义
3. 复用 package 模板建包
   - 输入：标准模板
   - 输出：新领域 workflow package 初始结构
4. 写 domain contracts
   - 输入：全局 envelope、局部扩展规则
   - 输出：domain-specific contract 文档/结构
5. 接入现有 route / bundle / activation
   - 输入：现有查询与执行链路
   - 输出：可被路由命中的新包入口
6. 写 domain validator
   - 输入：validation plan 与 validator result 结构
   - 输出：至少一个 domain validator
7. 选一个固定主任务
   - 输入：领域最小可验证场景
   - 输出：固定主任务定义
8. 加入 regression
   - 输入：现有回归矩阵
   - 输出：新增领域的 golden/negative/partial/handoff 用例

## 五、什么情况下说明“方法真的可复用”
以下条件同时成立时，说明不是只对第一条线有效：

1. 第二条领域线无需改系统层壳层
2. 第二条领域线能沿同一 package 模板落地
3. 第二条领域线能生成自己的 artifact
4. 第二条领域线能挂同一 validation / activation envelope
5. 第二条领域线能加入同一 regression 方法

### 不成立信号（任一出现即说明复用失败）
1. 为接入新领域而修改系统层主判断。
2. 为接入新领域而修改 envelope 顶层语义。
3. 新领域无法在固定主任务上形成可重复回归。
4. 新领域必须依赖口头规则才能运行。

## 六、M4 之后的正确推进方式
正确推进不是：
- 重新讨论 operating model
- 重新讨论 envelope 名字
- 重新讨论是否要最小上下文

正确推进是：
- 在已冻结骨架上复制第二条线
- 用第二条线验证复用性
- 再考虑是否需要更大范围扩域

## 七、M4 输出给下游的最小交接包
1. 准入检查清单模板（用于候选领域评审）。
2. 复用步骤的输入/输出说明（用于执行接入）。
3. 不成立信号与回退策略（用于避免隐式重构系统层）。
