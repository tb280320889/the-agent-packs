# 52 M4 上下文：第二领域线准入与可复用骨架

## 作用域
本文只回答：
1. 什么时候允许进入第二领域线
2. 进入时复制什么、替换什么
3. 怎样证明当前方法论真的可复用

## 一、第二领域线准入门槛
只有以下条件全部满足，才允许第二领域线进入：
1. 第一阶段主任务闭环已跑通
2. `wxt-manifest` 已稳定
3. `validator-core-output` 已稳定
4. `validator-domain-wxt-manifest` 已稳定
5. activation / bundle / artifact / handoff 顶层对象已冻结
6. Activation Result / Validation Plan 已固定
7. WXT 没有重大边界争议未解决

任何一条不满足，都不应开第二领域线。

## 二、第二领域线的正确进入方式
进入第二领域线时，应该做的是 **复制骨架**，不是重写系统。

### 必须复制的东西
- activation model
- blueprint frontmatter model
- package.yaml model
- artifact model
- validator 分层
- handoff model
- 里程碑推进法
- 文档包结构（指导文档 + 上下文文档）

### 只允许替换的东西
- domain 名
- blueprint nodes
- pack goals
- templates 内容
- domain validators
- route 条件

### 禁止重写的东西
- activation request 语义
- context bundle 顶层结构
- artifact descriptor 顶层结构
- handoff bundle 顶层结构
- completed / partial / handoff / failed 状态语义

## 三、复制式扩域模板

### 新领域线的最小进入包
建议仍按相同模式：
- 一个主 pack
- 一到两个薄横线
- 一个主 artifact
- 一个 core validator
- 一个 domain validator
- 一条固定主任务

### 新领域线的最小里程碑
仍按：
- M0 规则复核
- M1 节点与最薄入口
- M2 首个完整 pack
- M3 validator 与闭环
- M4 冻结与准入

## 四、证明“方法可复用”的判定
只有当第二领域线能在不改骨架的前提下成立，才能证明当前方法论真正通用。  
判断标准：
- 不需要新增顶层协议对象
- 不需要修改 artifact 类型表
- 不需要改状态枚举
- 不需要把 bounded context 放宽成宿主默认扫描
- 只通过替换领域内容就能跑通闭环

## 五、第二领域线候选选择规则
第一条新领域线建议只在以下候选中选一条：
- Tauri
- Axum

不要同时开第三条、第四条。

选择时优先考虑：
1. 域边界是否清楚
2. artifact 是否清楚
3. validator 是否容易写
4. 是否能挂现有横线
5. 是否不需要真实业务 app 才能验证

## 六、进入前的最后检查
若出现以下任何信号，应停止开新领域：
- 还在频繁改 schema
- 主任务虽然跑通，但 regression 不稳定
- wxt-manifest 仍频繁吞并相邻问题
- 还没有明确的 handoff 纪律
- 当前团队想靠新领域来“绕过”第一领域线未解决问题

## 七、本文结论
第二领域线的价值不在于覆盖更多，而在于证明：
- 当前方法论可复制
- 当前骨架可复用
- 后续扩展是“换内容，不换骨架”
