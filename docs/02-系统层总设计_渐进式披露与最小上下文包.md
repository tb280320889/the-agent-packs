# 02 系统层总设计：渐进式披露与最小上下文包

## 文档类型
系统层主文档

## 作用域
这是整套项目最重要的母版之一。  
它负责保存 **agent 的渐进式披露系统层设计**，必须长期稳定，不能在后续里程碑中被降级为“只是某个上下文实现细节”。

## 本文解决什么
1. 整个系统为什么不是普通文档库。
2. Git / Markdown / SQLite / MCP / Agent consumption 之间如何闭合。
3. Agent 为什么只能拿最小上下文包。
4. L0/L1/L2/L3 如何按阶段展开。
5. Blueprint Query MCP 的角色是什么。
6. 这套系统如何防止 agent 读一堆文档后自行猜测。

## 非作用域
- 不定义某个具体 pack 的业务规则
- 不定义某个具体 artifact 的模板细节
- 不定义领域树的全部内容

---

## 一、系统的核心判断

整套系统应固定为：

**Git 作为文档源真相层，Markdown + frontmatter 作为蓝图内容层，SQLite 作为结构化索引与轻量检索层，MCP 作为薄查询层，Agent 只通过 MCP 获取最小上下文包。**

这不是“方便实现”的建议，而是系统层设计本身。

---

## 二、四层系统闭合

### 1. 文档系统
负责存：
- L0/L1/L2/L3 分层 markdown
- frontmatter 元数据
- 目录、命名、路径规则
- 节点正文与摘要模板

它回答的问题：
- 某个领域增强包是什么
- 什么时候进入
- 依赖哪些横线
- 什么时候停止
- 什么时候应该交接

### 2. 存储系统
负责存：
- 节点索引
- 关系图
- 摘要
- 构建状态
- 查询缓存

它的本质不是业务数据库，而是 Blueprint 的结构化索引层。

### 3. 查询系统
负责：
- route_query
- expand_node
- read_node
- build_context_bundle
- validate_blueprint_graph
- rebuild_index

它回答的问题：
- 当前任务最适合哪个 pack
- 哪些节点是必须带的
- 哪些节点只应延迟展开
- 推荐哪些 validator
- 推荐哪些 artifacts

### 4. Agent 消费接口
负责：
- 给总编排一个稳定入口
- 给子领域编排一个固定展开协议
- 给执行态 agent 一个只读必要信息的接口

它的本质不是宿主管理器，而是面向 enhancement pack 的消费控制层。

---

## 三、为什么必须是 Git 管源、SQLite 管查

### Git 的职责
- 文档真相源
- PR / review
- 版本
- 回滚
- 变更审查

### SQLite 的职责
- 稳定查询
- 关系过滤
- 受控展开
- 轻量缓存
- 可选 FTS5

### 结论
**Git 管源，SQLite 管查。**  
不能把二者混成一个东西，也不能让 agent 直接把 Markdown 当路由索引。

---

## 四、为什么必须有 Blueprint Query MCP

Blueprint Query MCP 不是附属实现。  
它是把 Blueprint 图谱能力暴露给 agent 的 **唯一受控入口层**。

### 它必须承载三类原语
1. **resources**
   - 暴露节点内容
   - 暴露关系列表
   - 暴露 bundle 结果

2. **tools**
   - 暴露路由、展开、读取、构包、校验、重建能力

3. **prompts**
   - 暴露固定展开模板与固定决策模式

### 这样做的意义
- agent 不自由搜文档
- agent 不自由拼装上下文
- agent 只在固定接口面上工作
- 查询、展开、裁剪、bundle 生成都可测试、可替换、可校验

---

## 五、渐进式披露：L0/L1/L2/L3 的固定职责

### L0：领域入口层
只写：
- 领域定义
- 何时进入
- 何时不要进入
- 必带横线
- 推荐下一跳

### L1：子领域编排层
只写：
- 子领域目标
- 输入输出
- 依赖
- 常见分支
- 升级条件
- 停止条件

### L2：执行策略层
只写：
- 执行步骤
- 配置点
- 工具调用点
- validator 前置条件

### L3：边缘与升级层
只写：
- edge cases
- 平台差异
- 深度排障
- 回退路径

---

## 六、agent 真正怎么用这套系统

### 阶段 1：总编排
总编排不直接搜索 Markdown。  
它只允许：
- route_query(level=L0)
- route_query(level=L1, domain=主域)
- build_context_bundle(...)

### 阶段 2：子领域编排
拿到 bundle 后：
- 主节点默认只读 L1 摘要
- 必带横线只读 L1 摘要
- 不默认展开 L2/L3

### 阶段 3：执行态
只有进入执行态，才允许补 1~2 个必要 L2。

### 阶段 4：validator
validator 失败时：
1. 先看是否缺依赖
2. 再决定是否 expand_node
3. 最后才开 L3

### 项目态事实的例外处理
如果缺的是 **项目态事实**，不再继续搜蓝图，而改走：
- 代码搜索
- 配置读取
- 构建日志
- runtime resources

---

## 七、最小上下文包：为什么是系统核心

最小上下文包的目标不是“压缩文本”，而是控制 agent 的认知边界。

### 必须遵守的四条原则
1. **Summary first**  
   默认传摘要，不默认传正文。

2. **Required first**  
   先带主节点与 required 节点，不先带 may_include。

3. **Deferred by default**  
   L3、深层 L2 默认只给延迟展开清单。

4. **Minimality over completeness**  
   只给当前步完成任务所需的最小闭包，不追求一次给全。

### 统一结构
最小上下文包至少应包含：
- `main`
- `required`
- `execution_children`
- `deferred`

它存在的目的，是保证 agent 永远不能因为“上下文过多”而越界推理。

---

## 八、禁止事项

### 绝对禁止
- 让 agent 直接自由搜索 Blueprint Markdown
- 让全文正文成为 route 主索引
- 在 route 阶段默认深读 L2/L3
- 因为上下文不足就默认递归扩图
- 用“模糊相似文档”替代明确路由
- 用 bundle 装载一堆未来可能有用但当前不需要的节点

### 原因
这些行为会把系统退化成：
- 普通文档库
- 模糊检索器
- 上下文垃圾桶
- 无法测试的 prompt 系统

---

## 九、首期必须证明的不是“很多功能”，而是这条链成立

首期必须证明：

**Blueprint markdown -> compiler -> SQLite index -> Blueprint Query MCP -> minimal context bundle -> workflow pack -> validator -> activation result / handoff**

这条链一旦成立，系统层母版才算真的落地。

---

## 十、后续任何里程碑都不得破坏的稳定判断

1. 系统是 agent-driven，不是自治平台
2. Blueprint 是图谱，不是自由文档集
3. MCP 是受控查询层，不是旁路工具箱
4. bundle 是最小上下文闭包，不是全文打包
5. L3 是升级层，不是默认阅读层
6. 项目态事实优先走运行时/代码/配置资源，不继续滥扩蓝图
