# 42 M3 上下文：测试矩阵与固定主任务闭环

## 文档类型
单里程碑所需上下文文档

## 作用域
定义 M3 必须落下的测试矩阵与固定主任务。  
没有这份文档，M3 会退化成“跑过一次看起来没问题”。

## 本文必须回答的问题
1. 固定主任务是什么。
2. golden case 应是什么。
3. negative / partial / handoff 应如何覆盖。
4. regression 至少包含哪些类。
5. 闭环成功的判断标准是什么。

---

## 一、固定主任务

首期固定主任务建议为：

**review WXT manifest permissions for browser store submission**

### 选择原因
- 命中 `wxt-manifest`
- 可挂 `security-permissions`
- 可挂 `release-store-review`
- artifact 好定义
- validator 好写
- route / bundle / handoff 都容易观察

---

## 二、主任务的预期闭环

### 输入
- activation request
- bounded context 中有 manifest / config 相关片段
- 明确目标域为 WXT 或可由 task 推断

### 预期 route
- 主节点：`L1.wxt.manifest`
- 必带横线：`L1.security.permissions`、`L1.release.store-review`

### 预期 bundle
- `main`：L1.wxt.manifest 摘要
- `required`：security / release 两条横线摘要
- `execution_children`：L2.wxt.manifest.permissions-review
- `deferred`：L3.wxt.manifest.edge-cases 等

### 预期 artifact
- `manifest-review.md`

### 预期 validators
- `validator-core-output`
- `validator-domain-wxt-manifest`

### 预期最终结果
通常应为：
- `completed`，若信息充分且无阻塞失败
- `partial`，若上下文不足但已有局部结论
- `handoff`，若当前审查结束且必须交给 security / release 继续

---

## 三、最低测试矩阵

### A. Golden cases
1. **完整成功**
   - route 正确
   - bundle 最小
   - artifact 结构完整
   - validators 通过
   - activation result = completed

2. **有警告但可继续**
   - domain validator 出 warn
   - activation result = partial 或 completed（取决于 severity policy）

### B. Negative cases
1. **错误 target_pack 不应被模糊 hint 覆盖**
2. **anti_trigger 命中时不应强进包**
3. **无 activation request 时不应进入领域执行逻辑**
4. **bundle 不应默认塞入大量 L2/L3**

### C. Partial cases
1. **上下文不足，但主域明确**
2. **只能给出局部 manifest 风险说明**
3. **需要用户补 manifest / config 片段**

### D. Handoff cases
1. **manifest 已审完，但需要 security 深审**
2. **manifest 已审完，但需要 release/store 检查**
3. **handoff bundle 中 carry_context 必须存在**

---

## 四、回归集最低组成

至少包含：
- `fixtures/pass/`
- `fixtures/fail/`
- `tests/routing/`
- `tests/validation/`
- `tests/regression/`

### regression 必须覆盖
- 路由正确性
- bundle 最小化
- core validator
- domain validator
- activation result 状态计算
- handoff bundle 形状

---

## 五、闭环成功的定义
以下条件全部满足时，M3 才算真正跑通：

1. 主任务能从 request 进入 route
2. route 能选中 `wxt-manifest`
3. bundle 能构造最小闭包
4. pack 能生成主 artifact
5. validators 能输出结构化结果
6. activation result 能统一承载结果
7. 至少一类 regression 已固定

---

## 六、M3 绝不能省略的测试
最容易被省掉但必须保留的有三类：
1. **上下文不足的 partial**
2. **需要显式 handoff 的 case**
3. **route 不该被模糊 hint 覆盖的 negative case**

---

## 七、M3 冻结对象清单（测试与闭环视角）

### 冻结主任务
- `review WXT manifest permissions for browser store submission`

### 冻结 route 预期
- 主节点：`L1.wxt.manifest`
- 必带横线：`L1.security.permissions`、`L1.release.store-review`

### 冻结 bundle 形状
- 必须包含：`main`、`required`、`execution_children`、`deferred`
- 默认最小化原则：不默认深读大量 L2/L3

### 冻结 validators
- `validator-core-output`
- `validator-domain-wxt-manifest`

### 冻结结果状态与优先级
- 状态集合：`completed` / `partial` / `handoff` / `failed`
- 优先级：`failed > handoff > partial > completed`

### 冻结最小回归覆盖面
- Golden：完整成功、warning 可继续
- Negative：target_pack 优先、anti_trigger、生效前请求校验、bundle 最小化
- Partial：上下文不足但主域明确
- Handoff：跨包交接且 `carry_context` 必须存在
