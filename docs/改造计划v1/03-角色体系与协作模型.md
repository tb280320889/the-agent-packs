# 03 角色体系与协作模型

## 文档类型
系统层主文档

## 作用域
定义改造计划 v1 采用的角色体系，并把用户在对话中提出的“项目内部维护 agent + 两个子 agent + 用户侧角色”沉淀为正式结构。

## 一、内部角色

### 1. 项目内部维护 agent
主职责：
- 回答用户关于项目本身的问题
- 判断用户请求属于答疑、改造、吸收、还是执行
- 在内部把工作路由给合适子 agent
- 维护共享上下文、冻结对象与文档边界

### 2. 迭代开发子 agent
主职责：
- 执行项目内部的重构、升级、扩展开发
- 维护 route / bundle / registry / package template 等结构骨架
- 在 GSD 任务记录中负责任务级推进与收口

### 3. 文档型 agent
主职责：
- 接收 skill、MCP、llm.txt、外部最佳实践资料
- 做语义归类、规范映射、文档化与资产落位
- 把“外部资料”转化为“本项目可被 route / read / bundle 消费的正式资产”

## 二、外部角色

### 1. 用户
主职责：
- 阅读 README，快速知道项目定位、安装方式、基本能力边界
- 给出业务目标、疑问或要搜集的外部知识方向

### 2. 用户侧外部 agent
主职责：
- 读取本项目 README、文档与 MCP 能力说明
- 代表用户在本地理解本项目
- 搜集外部世界知识源并整理为可候选纳入的资料

## 三、协作边界

### 项目内部维护 agent 不做什么
- 不直接替用户执行外部世界的所有采集任务
- 不绕过文档吸收管道把资料直接塞入主文档

### 迭代开发子 agent 不做什么
- 不在没有冻结边界时直接扩展新领域线
- 不把“为了实现方便”当作修改全局 envelope 的理由

### 文档型 agent 不做什么
- 不把外部资料原样复制进系统主文档
- 不替代 route / registry / validator 的结构约束

## 四、与现有多-agent 模型的对应
- 项目内部维护 agent：更接近总入口 + Planner / Handoff 组合体
- 迭代开发子 agent：更接近 Executor
- 文档型 agent：更接近 Handoff + Knowledge Maintainer
- 后续若需要验证子 agent，可继续沿用 Verifier 角色

## 五、必须落地的共享对象
为了让这套角色体系真正可执行，任何一次接力都必须留下：
- GSD 任务记录
- Context Snapshot
- Handoff
- 冻结对象或风险说明
