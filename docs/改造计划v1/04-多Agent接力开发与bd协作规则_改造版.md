# 04 多Agent接力开发与 GSD 协作规则（改造版）

## 作用域
本文件只定义改造计划 v1 期间，多 agent 如何协作、如何拆 GSD 任务项、如何共享改造上下文。

## 一、核心原则
- GSD 进度追踪/文档系统仍是唯一任务真相源
- 正式边界与结构规则仍落在文档中
- 交接必须通过 Snapshot / Handoff 落地
- 不允许把“新的索引模型”“新的命名规则”只停留在对话里

## 二、推荐分工

### 项目内部维护 agent
- 负责接收需求
- 负责 GSD 任务项初步归类
- 负责决定交给开发子 agent 还是文档型 agent

### 迭代开发子 agent
- 领取实现型 GSD 任务项
- 输出开发指导、上下文文档、结构变更说明

### 文档型 agent
- 领取资料吸收 / 治理型 GSD 任务项
- 输出知识映射、吸收规则、资产落位文档

## 三、GSD 任务项拆分建议
- `epic`：整套改造计划 v1
- `task`：单一里程碑文档包、注册表设计、route 分层设计、迁移演练
- `bug`：发现现有实现与目标模型冲突之处
- `chore`：目录初始化、同步、文档整理

## 四、改造阶段必须额外记录的内容
除了原有 GSD 任务项状态外，还必须记录：
- 当前设计属于“冻结 / 草案 / 演练 / 已迁移”哪一类
- 是否影响现有 M0~M4 正式主线
- 是否触及 breaking 讨论

## 五、禁止事项
- 不建立 GSD 任务项就直接大改文档目录
- 不在无 Snapshot / Handoff 的情况下切换改造阶段负责人
- 不把“新架构草案”误写进现有正式主线而不标明边界
