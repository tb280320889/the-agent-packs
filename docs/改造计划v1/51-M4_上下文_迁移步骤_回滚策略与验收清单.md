# 51 M4 上下文：迁移步骤、回滚策略与验收清单

## 文档类型
单里程碑所需上下文文档

## 一、建议迁移顺序
1. 先冻结角色模型与边界
2. 再冻结节点分类与 route 分层规则
3. 再设计 package 注册表与命名治理
4. 再设计文档吸收管道
5. 最后做实现迁移与第二领域线演练

## 二、迁移实施阶段表

### 阶段 0：输入冻结确认
- 输入：`00~05` 共享文档、`50`、`51`、M3 快照与 handoff、`workflow-packages/registry.json`、`internal/registry/registry.go`。
- 目标：确认 M4 只承接前序冻结对象，不重新打开 M0~M3 设计。
- 产出：输入边界表、上游依赖表、非目标清单。
- 失败信号：发现前序冻结对象缺失、命名空间真相源不清、handoff 不足以继续执行。

### 阶段 1：迁移差异建模
- 目标：建立 `现状对象 -> 目标对象 -> 依赖前序冻结对象 -> 风险 -> 兼容策略` 的差异表。
- 最小覆盖范围：
  - 路由候选空间
  - 主域与横线 attach 关系
  - 注册表字段与准入校验
  - 资料吸收后的资产落位
  - validator / handoff 消费链路
- 产出：迁移差异表、整改对象列表。
- 失败信号：必须依赖别名、硬编码或临时说明才能成立，说明差异建模不完整。

### 阶段 2：兼容期护栏
- 目标：定义哪些现有行为可暂时保留，哪些必须立即禁止，哪些仅做观测。
- 兼容保留项示例：对既有已注册 package 的兼容检索。
- 禁止兼容项示例：横线 capability 以 direct route 方式进入第一轮主竞争、裸名 package 放行、未映射资产直接落正式目录。
- 观测项示例：旧 alias 触发频率、历史包命名整改清单、旧资产归属补齐清单。
- 产出：兼容规则表、禁止项列表、观测点列表。

### 阶段 3：回滚层级设计
- 目标：把回滚从“整体回退”拆成可定位的层级回退。
- 产出：文档层、注册表层、路由层、资产层回滚表。
- 失败信号：出现失败时只能整体撤回，无法限定影响面。

### 阶段 4：第二领域线准入演练
- 目标：验证第二领域线在演练态下能走通“注册 -> 吸收 -> 路由 -> 校验 -> handoff”链路。
- 演练对象：模拟主域、模拟 capability line、模拟 package 注册项、模拟知识资产。
- 产出：演练场景表、正例结果、负例结果、阻塞条件。
- 失败信号：只有成功路径，没有非法输入拒绝路径。

### 阶段 5：最终验收与收口
- 目标：把文档、实现差异、兼容护栏、演练结果收束为统一验收结论。
- 产出：验收矩阵、收口判定、后续 bead 建议、handoff。
- 失败信号：下一位 agent 仍无法直接判断“可依赖什么 / 不要做什么”。

## 三、迁移差异最小检查项
- `global orchestrator` 是否先于 `domain orchestrator` 做主域判断。
- `domain orchestrator` 是否只在主域内选择 workflow package，并决定 capability attach。
- capability package 是否仍由注册表中的 `category + visibility_scope + activation_mode` 控制可见性。
- 资产是否仍满足 `source_uri -> target_scope -> target_ref -> 落位` 完整链路。
- validator / handoff 是否消费最小上下文包，而不是退回全文自由搜索。

## 四、回滚策略

### 文档层回滚
- 触发条件：迁移规则无法自洽、阶段顺序与前序冻结对象冲突。
- 回滚动作：退回“仅冻结前序规则，不形成执行迁移顺序”的状态，补 bead 与快照说明缺口。
- 保留对象：M0~M3 已冻结对象全部保留，不受影响。

### 注册表层回滚
- 触发条件：`name / category / visibility_scope / activation_mode / canonical_blueprint_node` 无法稳定支撑迁移或准入演练。
- 回滚动作：退回“先补 registry 差异与整改清单，不进入实现迁移”。
- 禁止动作：不得为了通过演练临时放宽裸名、alias、reserved 规则。

### 路由层回滚
- 触发条件：横线 capability 重新进入第一轮主竞争，或候选空间无法按作用域收窄。
- 回滚动作：退回“仅保留文档与注册表冻结，不执行 route 行为迁移”。
- 保留要求：attach-only 语义必须保持冻结，不得被兼容期吞掉。

### 资产层回滚
- 触发条件：资产无法映射到已注册命名空间，或正式目录混入 backlog / reject 候选。
- 回滚动作：退回“候选池 + 手工映射”最小模式，未通过项转回 backlog。
- 禁止动作：不得在迁移层临时放行未归属资产。

## 五、兼容验证矩阵

### 文档与结构
1. 是否已有明确的总编排 / 主域编排 / 横线挂接三层模型。
2. 是否已有 package 注册表字段、命名规则与保留名约束。
3. 是否已有文档吸收标准管道与审查闸口。

### 行为与约束
4. 是否已禁止裸名 package。
5. 是否已把 capability package 限定为 attach-only 或 capability-scoped 可见。
6. 是否已明确兼容保留项、禁止兼容项、观测项。

### 资产与可追溯性
7. 是否能验证每项资产的 `source_uri -> target_scope -> target_ref -> 落位` 链路。
8. 是否能识别并回退未注册命名空间或无法映射的资产。

### 演练与收口
9. 是否已有第二领域线接入前的演练方案。
10. 是否已有通过 / 阻塞 / 拒绝三类验收结论与后续动作。

## 六、第二领域线准入演练清单

### 正例
- 正例 1：模拟第二主域注册后，可由 `global orchestrator` 正确识别主域，再由 `domain orchestrator` 选择主域 workflow package。
- 正例 2：模拟第二主域下的 capability attach，横线能力仅在主域确认后可见，不参与第一轮主竞争。
- 正例 3：模拟一份外部资料经 M3 管道映射到第二主域 package，能完成 `source_uri -> target_scope -> target_ref -> 落位` 校验。

### 负例
- 负例 1：尝试注册裸名 package，如 `security`、`manifest`，应被拒绝。
- 负例 2：尝试让 capability package 以 direct route 进入主竞争，应被拒绝。
- 负例 3：尝试把未映射资产直接落位到正式 package 目录，应被回退到 backlog 或 reject。
- 负例 4：尝试仅凭 alias 让历史对象越过候选空间约束，应被拒绝。

## 七、最终收口标准
- `可执行`：迁移顺序、前置依赖、失败信号、回滚点都已明确。
- `可回退`：文档层、注册表层、路由层、资产层都存在独立回滚路径。
- `可验证`：兼容验证与准入演练同时覆盖正例和负例，不靠口头判断通过。
- `可接力`：下一位 agent 可仅凭 bead、snapshot、handoff 与最小阅读集继续执行。
- `不越界`：M4 未修改顶层协议语义，未绕开注册表或资料治理管道。

## 八、验收结论类型
- `pass`：M4 文档已形成可执行迁移闭环，可进入实现迁移 bead。
- `pass-with-followup`：M4 主闭环成立，但仍有整改项需通过后续 bead 处理。
- `blocked`：存在前置冻结对象缺口或验证缺口，需先补上游交付。
- `reject`：当前迁移方案违反冻结边界，不能进入后续执行。
