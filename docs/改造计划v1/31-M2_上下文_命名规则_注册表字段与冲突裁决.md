# 31 M2 上下文：命名规则、注册表字段与冲突裁决

## 文档类型
单里程碑所需上下文文档

## 一、命名规则

### canonical name 总规则
- 每个 package 必须只有一个 canonical name，并作为注册表主键。
- canonical name 一旦冻结，不应被 alias 反向覆盖，也不应与保留名重叠。
- package 目录名、`package.yaml` 的 `name`、注册表 `name` 必须一致。

### 主域编排包
`<domain>-orchestrator`

### 主域 workflow package
`<domain>-<subdomain>`

### 横线 workflow package
`<capability-line>-<subdomain>`

### alias 规则
- alias 只用于检索、兼容旧称或触发提示，不作为身份真相源。
- alias 不得与任何已存在 canonical name 冲突。
- alias 不得绕过 `visibility_scope` 或 `activation_mode`，只能在合法候选空间内生效。

## 二、命名空间判断

### 主域 package
- `category = workflow`
- `domain` 必填，且必须指向已知主域。
- `visibility_scope` 默认是 `domain-scoped`。
- 若承担主域内主入口职责，可通过 `canonical_blueprint_node` 挂回对应 `workflow-entry`。

### capability package
- `category = capability`
- `domain` 为空或指向被附挂主域之外的独立能力线；其主归属由 `capability line` 决定。
- `visibility_scope` 默认是 `capability-scoped`。
- `activation_mode` 默认是 `attach-only`，不得作为第一轮主竞争候选。

### orchestrator package
- `category = orchestrator`
- 用于表达全局总编排或主域编排入口。
- `subdomain` 可为空，但 `canonical_blueprint_node` 必须稳定可引用。

## 三、禁止的裸名
以下名称默认禁止直接作为 package 名称：
- `release`
- `security`
- `auth`
- `payment`
- `manifest`
- `review`

### 保留名解释
- `release`、`security`、`auth`、`payment` 只能作为 capability line 前缀的一部分出现，例如 `security-permissions`。
- `manifest`、`review` 不能单独表达 package 身份，必须放回具体主域或能力线上，例如 `wxt-manifest`、`release-store-review`。
- 任何新增高复用词若可能跨多个主域复用，应先进入保留名审查，再决定是否允许注册。

## 四、注册表最小字段
- `name`
- `kind`
- `domain`
- `subdomain`
- `category`（orchestrator / workflow / capability）
- `canonical_blueprint_node`
- `visibility_scope`
- `activation_mode`
- `aliases`
- `reserved`

### 字段语义
- `name`：package canonical name，同时也是目录名与对外稳定标识。
- `kind`：对象类型；当前阶段至少区分 `workflow-package`，后续扩展不能改变 `name` 语义。
- `domain`：主域归属；主域 package 必填，capability package 不以此字段充当主身份。
- `subdomain`：当前 package 的细分职责；与 `domain` 或 capability line 共同组成 canonical name。
- `category`：决定对象属于 `orchestrator / workflow / capability` 哪一层结构角色。
- `canonical_blueprint_node`：注册表与 Blueprint 索引节点的稳定绑定；缺失时不应注册为正式 package。
- `visibility_scope`：决定节点默认可出现在哪一层候选空间。
- `activation_mode`：决定节点可 direct route 还是只能 attach。
- `aliases`：兼容旧称、触发词或检索词列表，不得与任何 canonical name 冲突。
- `reserved`：显式标记名称或条目是否处于保留治理范围，供准入校验和迁移脚本消费。

### 最小解释约束
- `category + visibility_scope + activation_mode` 必须足以判断一个 package 会不会进入第一轮主竞争。
- `canonical_blueprint_node` 缺失时，注册表不得把该对象当作正式可路由 package。
- `reserved = true` 的对象不能被当作普通 workflow package 直接新增或重名复用。

## 五、冲突裁决规则
当新增 package 时必须检查：
1. 名称是否重名
2. alias 是否撞已有 canonical name
3. 是否错误使用裸名
4. 是否误把 capability package 放进主域主竞争空间
5. 是否应复用已有 capability line 而非新建全局入口

### 建议裁决顺序
1. 先检查 `name` 是否与现有 canonical name 完全冲突。
2. 再检查 `name` 是否命中保留名或裸名禁用规则。
3. 再检查 `aliases` 是否撞现有 canonical name 或现有 alias。
4. 再检查 `category / visibility_scope / activation_mode` 组合是否与命名空间归属一致。
5. 最后检查该对象是否其实应复用既有 capability line、既有主域 subdomain，或既有 orchestrator。

### 裁决结果类型
- `allow`：允许注册。
- `rename-required`：语义可成立，但名称必须调整。
- `reuse-required`：不得新建入口，必须挂回已存在 capability line 或主域空间。
- `reject`：违反冻结规则、裸名禁令或候选空间规则，不能进入注册表。

## 六、注册表示例判断

### `wxt-manifest`
- 合法主域 workflow package。
- `category = workflow`，`domain = wxt`，`subdomain = manifest`。
- 可直接挂回 `L1.wxt.manifest` 一类稳定 Blueprint 入口。

### `security-permissions`
- 合法 capability package。
- `category = capability`，默认 `visibility_scope = capability-scoped`。
- `activation_mode` 应为 `attach-only`，不参与第一轮主竞争。

### `security`
- 非法裸名。
- 即使语义上属于能力线，也必须扩展成具体 capability package，不能直接注册为 workflow package。

## 七、给实现层与迁移层的约束
- route 层不得仅凭 alias 或 trigger 让 capability package 越过注册表候选空间规则。
- 迁移层若发现历史 package 不满足 canonical name 规则，应先登记为整改对象，而不是直接放宽 M2 规则。
- 新增 package 准入必须先过注册表校验，再进入 Blueprint / validator / handoff 设计。
