# 50 M4 迁移实施、兼容验证与准入演练 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M4 负责把前面三阶段的结构设计转化为可执行迁移计划，并通过演练验证第二领域线接入前的可行性。

## 本里程碑输入边界
- 只消费共享文档 `00~05`、M0~M3 已冻结对象、当前 M4 文档与已有注册表真相源。
- 直接承接 M1 已冻结的分层 route 约束、M2 已冻结的注册表与命名空间规则、M3 已冻结的资产吸收与落位规则。
- 迁移计划只能把既有冻结对象串成执行顺序，不得借迁移之名重写 `Activation Request`、`Routing Result`、`Context Bundle`、`Validation Plan`、`Validator Result`、`Activation Result`、`Handoff Bundle` 的顶层语义。
- 第二领域线演练只能用于验证准入路径，不得在本里程碑内默认扩张为正式常驻主域。

## 本里程碑目标
1. 定义从当前实现迁移到新索引模型的顺序
2. 定义兼容期与回滚策略
3. 定义第二领域线接入前的演练方法
4. 定义验收清单

## 本里程碑建议直接落地的共享产物
- M4 阶段快照：`docs/改造计划v1/context-snapshots/<date>-m4-migration-readiness.md`
- M4 交接文档：`docs/改造计划v1/handoffs/<bead-id>-handoff.md`
- M4 输出摘要：在文档中明确“迁移顺序 / 兼容期约束 / 回滚层级 / 准入演练 / 最终收口标准”，避免后续执行层自行补规则。

## 设计顺序
1. 先确认 M0~M3 冻结对象与输入边界，不允许在 M4 重新打开前序设计争议。
2. 再建立“现状实现 -> 目标分层骨架”的迁移差异表。
3. 再定义兼容期规则、失败信号与回滚层级。
4. 再定义第二领域线准入演练的正例、负例与通过条件。
5. 最后定义 Verifier 的验收矩阵、收口标准与交接要求。

## M4 必须回答的问题
- 现有实现从平铺闭环迁移到“总编排 -> 主域编排 -> 横线挂接 -> validator”骨架时，顺序为什么是当前顺序，前后置依赖分别是什么。
- 哪些对象只需要文档冻结，哪些对象需要进入实现对齐，哪些对象需要在兼容期继续保留观测点。
- 一旦注册表校验、候选空间裁剪、attach-only 语义或资产落位校验失败，应回滚到哪一层，谁负责阻断继续迁移。
- 第二领域线在尚未正式接入前，如何用模拟注册项、模拟资产与负例校验完成准入演练。
- M4 完成后，后续 agent 如何仅凭 bead、snapshot、handoff 与最小阅读集继续做实现迁移，而不再重新解释系统边界。

## 必须产出
1. 迁移步骤
2. 回滚策略
3. 兼容验证清单
4. 准入演练清单

## 分阶段实施要求

### 阶段 A：冻结输入面与迁移范围
- 明确可依赖的冻结对象、依赖实现与非目标。
- 输出迁移输入边界表与结构影响范围说明。
- 若发现前序里程碑交付缺口，必须记录 bead 或阻塞，不得脑补补齐。

### 阶段 B：迁移差异建模
- 对当前实现建立 `现状对象 -> 目标对象 -> 风险 -> 是否兼容保留` 的迁移差异表。
- 至少覆盖 `registry`、`route`、`bundle`、`validator`、`knowledge assets` 五类对象。
- 任何不满足 M1/M2/M3 冻结规则的历史对象，先标记为整改项，不得通过 M4 放宽规则。

### 阶段 C：兼容期与回滚纪律
- 定义兼容保留项、禁止兼容项、观测项三类对象。
- 定义文档层、注册表层、路由层、资产层四级回滚点。
- 定义失败信号：命名空间校验失败、attach-only 失效、横线误入主竞争、资产归属链路不完整、候选集越界。

### 阶段 D：准入演练与最终验收
- 通过模拟第二领域线接入验证“先注册、后资料、再 route、后 validator”的闭环。
- 同时覆盖正例与负例，不允许只跑成功路径。
- 输出验收矩阵、通过条件、收口标准与后续 handoff。

## 明确不产出
- 不直接在 M4 中扩张正式第二领域线主文档包。
- 不为了迁移便利而放宽裸名禁令、alias 约束、attach-only 语义或资产落位闸口。
- 不把兼容期当成永久双轨状态；兼容规则必须附带退出条件。
- 不把准入演练结果直接等同于正式生产接入结论。

## 完成标准
1. 改造不是纸面设计，而是可按顺序实施的迁移闭环。
2. 任意关键步骤若失败，都有清楚回退路径与阻断点。
3. 第二领域线接入前可先做演练验证，且演练同时覆盖正例与负例。
4. 后续执行层可直接消费 M4 输出，不需要重新解释迁移顺序与验收边界。

## 本阶段建议冻结的最小对象
- 迁移阶段最小集合：`输入冻结确认`、`迁移差异建模`、`兼容期校验`、`准入演练`、`最终收口`。
- 回滚层级最小集合：`文档层回滚`、`注册表层回滚`、`路由层回滚`、`资产层回滚`。
- 验收结果最小集合：`pass`、`pass-with-followup`、`blocked`、`reject`。

## Verifier 与 Handoff 要求
- Verifier 必须同时检查文档完整性、注册表约束、资产归属链路、候选空间约束与负例拒绝行为。
- Handoff 必须明确下一 bead、先阅读文档、先验证命令、不可触碰边界与剩余风险。
- 若 M4 产物仅完成计划层，不得标记为“已迁移”；应明确标记为“可执行迁移计划已冻结”。

## 给后续执行层的最小输出要求
- 实现迁移前，必须先核对 M4 定义的迁移差异表与兼容期约束。
- 第二领域线正式接入前，必须先通过 M4 定义的准入演练与负例校验。
- 任意执行层若想放宽规则，必须回到 bead + snapshot + handoff 体系重新立项，不得口头覆盖 M4 冻结边界。
