# 60 M5 实现迁移执行与差异整改 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M5 承接 M4 已冻结的迁移实施计划、兼容验证边界与收口结论，把“可执行迁移计划”下沉为真实实现整改与差异消解，确保 registry、route、knowledge assets、validator、handoff 的执行层行为与 M0~M4 冻结对象一致。

## 本里程碑输入边界
- 只消费共享文档 `00~05`、M0~M4 已冻结对象、M4 验收结论、当前 M5 文档与仓库内现有实现。
- 必须把 `docs/改造计划v1/52-M4_验收结果与收口结论.md` 视为输入边界，不得重新打开 M4 已通过的计划层结论。
- 允许修改执行层实现、测试、注册表样本、模板与校验逻辑，但不得重写顶层 envelope 语义。
- 若发现 M4 计划不足以支撑实现迁移，应记录 bead 并回流文档修订，不得在实现层口头绕过。

## 本里程碑目标
1. 形成可执行的迁移差异表与整改顺序
2. 完成 registry / route / assets / validator / handoff 的实现对齐
3. 落地兼容期护栏、观测点与退出条件
4. 形成可被 M6 直接消费的运行验证输入

## 本里程碑建议直接落地的共享产物
- M5 阶段快照：`docs/改造计划v1/context-snapshots/<date>-m5-execution-alignment.md`
- M5 交接文档：`docs/改造计划v1/handoffs/<bead-id>-handoff.md`
- M5 差异摘要：明确“已整改对象 / 待整改对象 / 兼容保留项 / 禁止兼容项 / 退出条件”。

## 设计顺序
1. 先把 M4 的迁移阶段转换为“实现对象清单”。
2. 再按风险与依赖顺序排列整改批次。
3. 再补齐测试、校验、观测与回滚脚手架。
4. 最后输出 M6 可直接消费的验证输入与未决项。

## M5 必须回答的问题
- 哪些实现对象已经符合 M1/M2/M3/M4 冻结规则，哪些仍需整改。
- 每个整改对象的顺序、依赖、风险、验证命令与回滚点是什么。
- 兼容保留项如何被限制在可退出边界内，而不是变成永久双轨。
- 资产链路、attach-only、候选空间裁剪在实现层如何被真正约束，而不只是文档声明。
- M6 如何在不重新分析全部设计文档的情况下直接接手运行验证与正式准入。

## 分阶段实施要求

### 阶段 A：差异清点
- 建立 `现状实现 -> 目标约束 -> 整改动作 -> 验证 -> 回滚` 的差异表。
- 至少覆盖 `workflow-packages/registry.json`、`internal/query/`、`internal/validator/`、资产目录与相关测试。

### 阶段 B：分批整改
- 按“注册表与身份 -> 候选空间与 route -> 资产归属与消费 -> validator 与 handoff”顺序推进。
- 每批必须伴随最小测试与回滚说明。

### 阶段 C：兼容落地
- 将兼容保留项转成可观测配置、校验或测试，不允许只写说明文字。
- 为每项兼容保留项定义退出条件与截止信号。

### 阶段 D：交接 M6
- 产出运行验证输入、已知风险、剩余待办与推荐验证顺序。
- 未完成整改项必须进入 bead，不得留在对话里。

## 明确不产出
- 不直接宣布第二领域线正式接入完成。
- 不用单个实现样本反向修改 M1~M4 冻结边界。
- 不把兼容观测脚手架误写成长期正式协议。

## 完成标准
1. M4 计划中的关键整改对象已映射到实际实现与测试。
2. 关键执行链路已具备可验证的实现约束，而非仅有文档约束。
3. 兼容保留项有退出条件且被显式记录。
4. M6 可以直接进入运行验证与正式准入阶段。

## 本阶段建议冻结的最小对象
- 差异状态最小集合：`not-started`、`in-progress`、`aligned`、`deferred`、`blocked`。
- 整改对象最小集合：`registry`、`route`、`asset`、`validator`、`handoff`、`tests`。
- 兼容结论最小集合：`keep-temporarily`、`remove-before-m6`、`reject`。

## 给后续里程碑的最小输出要求
- 对 M6：必须提供已整改对象清单、兼容保留项、待退出项、验证命令与风险说明。
