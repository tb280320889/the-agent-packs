# 21 M1 上下文：节点分类、作用域与可见性

## 文档类型
单里程碑所需上下文文档

## 作用域
定义改造计划 v1 需要新增的 Blueprint/索引分类字段。

## 一、建议新增字段

### `node_kind`
- `domain-root`
- `domain-orchestrator`
- `capability-root`
- `workflow-entry`
- `execution-strategy`

### `visibility_scope`
- `global`
- `domain-scoped`
- `capability-scoped`

### `activation_mode`
- `direct`
- `attach-only`

## 二、可见性原则
- `global` 节点可参与全局候选筛选
- `domain-scoped` 节点只有在主域确认后可见
- `attach-only` 节点不能成为当前轮主候选，只能由上游节点挂接

## 三、对横线能力的直接约束
`security`、`release`、`auth`、`payment` 一类能力默认：
- 作为知识域可以存在
- 作为 workflow package 可以存在
- 但默认 `activation_mode = attach-only`

## 四、目标结果
route 的“候选集”先由 `visibility_scope + activation_mode` 决定，再由 triggers / aliases / evidence 打分。

## 五、节点分类到注册表字段的映射建议
为避免 M2 重复发明字段，M1 应先给出最小映射关系：

| M1 判断对象 | 建议注册表字段 | 用途 |
|---|---|---|
| `node_kind` | `category` / `node_kind` | 区分 orchestrator、workflow、capability |
| `visibility_scope` | `visibility_scope` | 决定节点默认出现在哪一层候选空间 |
| `activation_mode` | `activation_mode` | 决定节点能否 direct route 或只能 attach |
| Blueprint 入口节点 | `canonical_blueprint_node` | 固定 package 与索引节点的归属关系 |

### 最小解释约束
- 若一个 package 对应 `activation_mode = attach-only`，则它不能作为第一轮主竞争候选。
- 若一个节点 `visibility_scope = domain-scoped`，则它必须在主域确认后才能进入候选池。
- 若一个对象没有稳定的 `canonical_blueprint_node`，则 M2 不应把它注册为正式 package。
