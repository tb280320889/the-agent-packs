# 10 M0 角色冻结与边界校正 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M0 只做一件事：**把改造计划 v1 的角色体系、系统边界、冻结面与不可动面先钉死。**

## 本里程碑目标
1. 冻结内部维护 agent / 迭代开发子 agent / 文档型 agent / 用户侧外部 agent 的角色模型
2. 冻结“总编排先于主域编排，主域编排先于横线挂接”的系统判断
3. 明确现有正式主线哪些对象完全不动
4. 明确改造计划 v1 只改结构，不改现有顶层 envelope 语义

## 本里程碑建议直接落地的共享产物
- M0 阶段快照：`docs/改造计划v1/context-snapshots/<date>-m0-role-freeze.md`
- M0 交接文档：`docs/改造计划v1/handoffs/<task-id>-handoff.md`
- M1 冻结摘要：直接写入 M0 交接文档，避免 M1 再回头拼装边界说明

## 明确不产出
- 具体 query 重构实现
- package 注册表实现
- 文档吸收流水线实现

## 完成标准
1. 后续里程碑不再争论角色定义
2. 后续里程碑知道哪些对象不能碰
3. M1 可以直接在冻结边界上设计分层索引模型

## M0 输出给 M1 的最小摘要
M0 完成后，至少要向 M1 明确以下结论：
- 项目内部维护 agent / 迭代开发子 agent / 文档型 agent / 用户侧外部 agent 的职责边界已冻结
- 改造计划 v1 只改结构层，不改正式主线顶层 envelope 语义
- `Blueprint 节点分类字段`、`route 候选集筛选逻辑`、`package 注册表` 属于兼容演化面
- 若后续设计需要触碰状态枚举、优先级、workflow package 基本模板，则必须停止并回到 breaking 讨论
