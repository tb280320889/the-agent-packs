# 01 通用改造方法论与增量迁移规则

## 文档类型
通用规则

## 作用域
适用于改造计划 v1 的所有里程碑。

## 一、总方法论

### 1. 增量替换，不整体推倒
当前 `docs/00~52`、`blueprint/`、`internal/`、`workflow-packages/` 已形成最薄闭环。改造计划只能在此基础上增量推进，不能以“架构更优”为由直接重写。

### 2. 先固定索引模型，再接更多包
新增 workflow package 之前，必须先把：
- 总编排
- 主域编排
- 横线挂接
- package 注册表
- 名称冲突治理

这些壳层固定下来。

### 3. 可见性先于命名
名字唯一只能解决“文件与对象不撞名”；真正决定 route 是否安全的是：
- 当前候选集是谁
- 当前上下文能看见哪些节点
- 横线节点是否只能 attach，不能 direct route

### 4. 横线能力不参与第一轮主竞争
`security`、`release`、`auth`、`payment` 一类横线能力，默认只能在主域明确后被挂接，不得默认参与全局主路由竞争。

### 5. 文档吸收必须走治理管道
skill、MCP、llm.txt、最佳实践文档、外部资料，不得直接散落入仓。
必须走：
- 候选收集
- 语义归类
- 文档映射
- 冻结审查
- 资产落位

### 6. 保持冻结面稳定
改造计划 v1 默认不得改动以下对象的顶层语义：
- Activation Request
- Routing Result
- Context Bundle
- Validation Plan
- Validator Result
- Activation Result
- Handoff Bundle

## 二、统一输入 / 输出模型

### 输入
每个里程碑只能消费：
1. 现有正式文档包的冻结对象
2. 本目录前序里程碑的冻结对象
3. 当前里程碑开发指导与上下文文档

### 输出
每个里程碑必须至少输出：
1. 结构化文档交付物
2. 冻结对象摘要
3. migration / compatibility 判断
4. Context Snapshot
5. Handoff

## 三、禁止事项
- 为了新架构方便，直接改写现有 Phase 1 顶层 envelope
- 在 route 阶段让横线能力直接和主域竞争
- 在没有注册表与命名治理前继续新增裸名 package
- 把外部资料直接复制进主文档而不做语义映射
- 只在对话里说“以后要改总编排”，不落到文档与 GSD 任务项

## 四、完成标准
当且仅当以下条件成立时，改造计划 v1 才算方法论成立：
1. 后续新增 package 不会因命名或索引范围冲突而误路由
2. route 已从平铺候选升级为分层候选
3. 横线能力的 attach-only 语义清楚
4. 文档吸收有固定管道而不是自由堆资料
5. 第二领域线接入前可通过准入与迁移演练进行验证
