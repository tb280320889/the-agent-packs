# 70 M6 运行验证、第二领域线准入实施与持续治理 开发指导

## 文档类型
开发里程碑开发指导文档

## 作用域
M6 承接 M5 的实现迁移与差异整改结果，把 v1 从“结构可执行、实现已对齐”推进到“可运行验证、可正式准入、可持续治理”的状态。

## 本里程碑输入边界
- 只消费共享文档 `00~05`、M4 验收结论、M5 已整改输出、当前 M6 文档与仓库实现状态。
- 第二领域线正式准入必须以 M5 已整改对象清单为前置输入，不得跳过实现对齐直接宣告接入。
- 允许扩展验证、准入脚本、持续治理规则与观测手段，但不得回头放宽 M0~M5 冻结对象。

## 本里程碑目标
1. 执行运行验证矩阵并给出正式结论
2. 完成第二领域线正式准入实施或明确阻塞条件
3. 清理必须退出的兼容保留项
4. 形成持续治理节奏与准入复用机制

## 本里程碑建议直接落地的共享产物
- M6 阶段快照：`docs/改造计划v1/context-snapshots/<date>-m6-runtime-governance.md`
- M6 交接文档：`docs/改造计划v1/handoffs/<bead-id>-handoff.md`
- M6 验收摘要：明确“运行通过 / 阻塞 / 部分通过”的正式结论。

## 设计顺序
1. 先执行 M5 提供的验证矩阵。
2. 再处理第二领域线正式准入。
3. 再退出临时兼容项。
4. 最后固定持续治理与新增领域线复用规则。

## M6 必须回答的问题
- 哪些验证项目已经通过，哪些仍 blocked，阻塞点在哪里。
- 第二领域线能否正式接入；若不能，缺的到底是实现、验证还是治理条件。
- 哪些兼容项必须在 M6 结束前退出，哪些可保留到后续治理节奏但必须继续观测。
- v1 在 M6 后如何复用到第三领域线或更多 workflow package，而不重新走一遍完整设计讨论。

## 分阶段实施要求

### 阶段 A：运行验证
- 覆盖注册表、route、资产归属、validator、handoff、第二领域线候选场景。
- 结论必须结构化，不允许只写“基本可用”。

### 阶段 B：正式准入实施
- 仅在验证通过且关键阻塞解除后进入。
- 第二领域线正式准入必须同时经过注册、route、asset、validator、handoff 全链路验证。

### 阶段 C：兼容退出与治理固化
- 清理 `remove-before-m6` 项。
- 固化新增领域线或 package 的复用准入模板、检查表与治理节奏。

## 明确不产出
- 不把仍 blocked 的场景包装成正式准入成功。
- 不把一次性的人工操作当成持续治理机制。
- 不在 M6 随意回改前序里程碑冻结对象。

## 完成标准
1. 有正式运行验证结论。
2. 第二领域线正式准入要么完成，要么被结构化阻塞。
3. 兼容退出状态清楚。
4. 持续治理机制可复用。

## 给后续阶段的最小输出要求
- 若仍需更长期治理，应以新里程碑或常规 roadmap 继续，不再回退为 M0~M6 混合阶段。
