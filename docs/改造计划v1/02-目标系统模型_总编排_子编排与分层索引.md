# 02 目标系统模型：总编排、子编排与分层索引

## 文档类型
系统层主文档

## 作用域
定义改造计划 v1 的目标系统模型，回答“为什么当前实现需要从平铺路由升级为分层索引”。

## 一、目标主链路

改造后的目标主链路固定为：

`用户 / 外部 agent -> 项目内部维护 agent -> global orchestrator -> domain orchestrator -> workflow package -> cross-cutting attach -> validator -> activation result / handoff`

这条链路中的关键变化不是新增更多工具，而是新增了两道受控索引面：
- `global orchestrator`
- `domain orchestrator`

## 二、三层索引面

### 1. 全局总编排层
负责：
- 识别主域
- 决定当前任务属于哪个主域空间
- 拒绝横线能力直接进入第一轮主竞争

### 2. 主域编排层
负责：
- 在主域内部选择 workflow package
- 决定需要挂接哪些横线能力
- 决定哪些横线只是 required，哪些只是 may_include

### 3. 横线挂接层
负责：
- 只在主域已确认时可见
- 只附挂，不抢主入口
- 提供安全、发布、身份、支付等能力线的摘要与后续 handoff 能力

## 三、为什么当前实现不够
当前实现存在以下结构性限制：
- `L1` 候选基本平铺，主域与横线能力过早出现在同一候选池
- `packNodeMap` / `nodeProfileMap` 是全局平铺映射
- `release` / `security` 作为知识域存在，但 route 时缺少作用域门槛

## 四、目标判断

### 必须成立
- package 名称全局唯一
- route 候选集按作用域收窄
- 横线能力默认 attach-only
- 文档吸收结果必须能挂回 domain / capability / package 命名空间

### 不允许成立
- 任何 `security-*` 或 `release-*` 包都可在全局主路由直接竞争
- 同一轮 route 同时比较“主域包”和“横线包”而没有作用域过滤
- 用 aliases 或 hints 替代可见性规则

## 五、后续里程碑不可破坏的判断
1. 总编排先于主域编排
2. 主域编排先于横线挂接
3. 横线能力默认 attach-only
4. 注册表是 package 身份真相源
5. Blueprint 节点分类与可见性决定 route 候选集，而不是仅靠模糊文本打分
