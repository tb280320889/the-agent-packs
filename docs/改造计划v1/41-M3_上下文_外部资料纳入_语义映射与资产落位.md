# 41 M3 上下文：外部资料纳入、语义映射与资产落位

## 文档类型
单里程碑所需上下文文档

## 一、外部资料候选类型
- skill 说明
- MCP server / resource / tool 文档
- llm.txt
- 官方最佳实践
- 可复用模板与规则清单

## 二、候选资料最小元数据
- `source_type`：资料类型，取值必须落在 M3 候选类型中。
- `source_uri`：原始来源地址、仓库路径或本地来源说明。
- `captured_at`：纳入候选池的时间。
- `owner`：本次负责吸收的 bead 或 agent。
- `trust_level`：`official / project-local / third-party / unknown`。
- `target_scope`：`system / domain / capability / package / reject`。
- `target_ref`：目标归属引用；若是 package，则必须指向注册表 canonical name。
- `disposition`：`candidate / mapped / frozen / backlog / rejected`。

### 元数据约束
- 缺少 `source_type`、`source_uri`、`target_scope`、`disposition` 的候选，不得进入正式审查。
- `target_scope = package` 时，`target_ref` 必须命中 `workflow-packages/registry.json` 中的 canonical name。
- `target_scope = capability` 时，`target_ref` 必须命中已冻结 capability line，不得从资料文本临时发明。
- `target_scope = reject` 时，必须记录拒收原因，避免重复吸收。

## 三、标准吸收流程
1. 收集候选
2. 做语义去重
3. 判断属于：
   - 系统层
   - 主域知识
   - 横线能力知识
   - workflow package 规则
   - 不纳入
4. 做规范映射
5. 形成正式文档资产或候选 backlog

### 各步骤最小要求

#### 1. 收集候选
- 只允许进入候选池，不允许直接写入主文档、Blueprint 正文或 package README。
- 每条候选必须附带最小元数据。

#### 2. 语义去重
- 对候选内容做主题、适用范围、约束来源三维去重。
- 若内容与已冻结资产重复，只保留更高可信度来源，并将其余来源挂为旁证。

#### 3. 归类判断
- `system`：影响系统级协作、协议、通用模式，但不绑定单一 package。
- `domain`：绑定某个主域，如 `wxt`。
- `capability`：绑定某条横线能力，如 `security`、`release`。
- `package`：只对已注册 package 生效，如 `wxt-manifest`。
- `reject`：与当前冻结对象冲突、无稳定来源、或无法映射到已注册命名空间。

#### 4. 规范映射
- 将外部表述转换为本项目内部语义，不保留“原样复制即完成”的路径。
- 明确该资料对应的是规则、模板、测试约束、合同片段、还是阅读指引。

#### 5. 冻结审查与落位
- 通过命名空间校验、非目标检查、落位检查、可消费性检查后，才能形成正式资产。
- 未通过者进入 backlog 或 reject，不得先落位后补审查。

## 四、语义映射规则

### 系统层
- 进入系统层文档或共享协作层文档。
- 不得借系统层资料绕过 M2 注册表或 M1 分层路由规则。

### 主域知识
- 必须绑定已知主域。
- 若主域尚未注册或未冻结，只能进入 backlog，不得先生成正式资产。

### 横线能力知识
- 必须绑定已冻结 capability line。
- 横线知识默认仍是 attach-only 辅助知识，不得被改写为全局主路由入口。

### package 规则
- 必须绑定注册表中的 canonical package name。
- 只允许补强 package 合同、模板、tests、README 或 handoff 约束，不得反向改变 package 身份。

### reject
- 满足以下任一条件时拒收：
  - 无法追溯来源
  - 与冻结对象冲突
  - 试图新增未注册命名空间
  - 只是营销性或泛泛最佳实践，无法转为项目可消费资产

## 五、落位原则
- 系统层资料进入系统层文档，不进入 package README
- 主域资料进入对应主域知识节点或主域文档
- 横线资料进入 capability 空间
- package 专属资料进入 package 合同、模板、tests 或说明文档

### 目录与对象落位约束
- 系统层资产：进入 `docs/改造计划v1/` 或后续共享知识目录中的系统层区域。
- 主域资产：进入对应主域文档或可被 Blueprint 绑定的主域知识节点。
- capability 资产：进入对应 capability 空间，且不得伪装成主域 package。
- package 资产：进入 `workflow-packages/<canonical-name>/` 下的说明、模板、tests 或合同位置。
- backlog：只记录在 bead、snapshot、handoff 或专门候选清单中，不充当正式知识资产。

## 六、文档型 agent 的职责边界
- 负责候选收集、语义去重、规范映射、资产落位建议。
- 负责把外部资料转为本项目可消费的正式资产描述。
- 不负责绕过注册表新增 package。
- 不负责改写 Activation Request、Routing Result、Context Bundle 等顶层冻结对象。

## 七、审查闸口
- 命名空间闸口：是否命中已注册 `domain / capability / package`。
- 去重闸口：是否与现有资产重复或冲突。
- 非目标闸口：是否试图越界改动 M0/M1/M2 冻结对象。
- 落位闸口：是否进入正确目录与对象类型。
- 可消费性闸口：是否能被 route / read / bundle / handoff 间接消费，而不是纯资料堆积。

## 八、给 M4 与后续执行层的约束
- M4 迁移时必须检查每项资产的 `source_uri -> target_scope -> target_ref ->落位` 链路是否完整。
- 若发现资产无法映射到已注册命名空间，应先回退到 backlog，而不是在迁移层临时放行。
- 后续若新增主域或 capability line，必须先完成 M2 注册表扩展，再允许 M3 吸收该空间资料。

## 九、禁止事项
- 把外部资料原样复制为 Blueprint 正文
- 不做语义映射就新增新的 capability 名称
- 用外部资料绕过既有注册表与命名规则
- 让 backlog 候选伪装成正式资产
- 让 package README 承担系统层或 capability 层规则真相源
