# 61 M5 上下文：迁移差异表、整改顺序与兼容落地

## 文档类型
单里程碑所需上下文文档

## 一、M5 的核心判断
- M5 不是重新讨论架构，而是把 M4 冻结的计划转成实现对齐动作。
- M5 的真相源依然是注册表、分层路由、资产归属链路与最小上下文消费原则。
- 若实现层与文档层冲突，先记录差异并按 bead 处理，不允许实现层直接覆盖文档冻结对象。

## 二、迁移差异表最小字段
- `object_type`：`registry / route / asset / validator / handoff / tests`
- `object_ref`：对象路径、模块名或稳定引用。
- `target_constraint`：对应的冻结规则或目标约束。
- `current_state`：当前实现状态摘要。
- `gap`：当前差距。
- `remediation`：整改动作。
- `validation`：验证命令、测试或检查方式。
- `rollback_point`：失败后回退位置。
- `status`：`not-started / in-progress / aligned / deferred / blocked`

## 三、整改顺序
1. 先整改注册表真相源与身份校验。
2. 再整改 route 候选空间、主域选择与 capability attach。
3. 再整改资产归属、落位与消费链路。
4. 再整改 validator、handoff 与最小上下文消费约束。
5. 最后整理兼容保留项、测试与退出条件。

## 四、兼容落地规则

### keep-temporarily
- 仅用于已知历史对象仍需短期过渡的情况。
- 必须有观测点与退出条件。
- 不得突破 attach-only、裸名禁令或未归属资产闸口。

### remove-before-m6
- 必须在 M6 正式运行验证前移除。
- 若未移除，不得宣告进入“正式准入实施”。

### reject
- 与冻结规则冲突或无法验证的兼容项直接拒绝。
- 不得因“历史原因”长期挂起为默认保留。

## 五、M5 的最小验证面
- 注册表校验与样本一致性。
- route 候选空间与 attach-only 行为。
- 资产 `source_uri -> target_scope -> target_ref -> 落位` 链路。
- validator / handoff 是否只消费最小上下文包。
- 兼容保留项是否具备退出条件与测试覆盖。

## 六、M5 对 M6 的交付接口
- 已整改对象清单。
- 待退出兼容项清单。
- 运行验证命令集合。
- 第二领域线正式准入前的阻塞项与风险。

## 七、禁止事项
- 以“先跑通再说”为由跳过差异表登记。
- 让 deferred 项混入 aligned 清单。
- 用人工口头说明替代验证命令、测试或回滚点。
