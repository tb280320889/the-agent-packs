# 42 M3 上下文：测试矩阵与闭环验收

## 作用域
本文只回答：
1. 第一阶段至少要测哪些 case
2. 哪些 case 是 golden path
3. 哪些 case 必须 fail / partial / handoff
4. 什么叫“主任务闭环已成立”

## 一、测试矩阵设计原则
- 只测与当前闭环直接相关的 case
- 先测 route / artifact / validator 的最小闭环
- 不为“未来也许有用”的 edge cases 提前扩矩阵
- 每个 case 都必须有明确期望状态

## 二、首批必须存在的 case

| Case ID | 类型 | 输入概况 | 期望主 pack | 期望状态 |
|---|---|---|---|---|
| G01 | Golden | manifest + permissions + store submission | wxt-manifest | completed 或 partial（取决于 release 上下文是否充足） |
| G02 | Golden | content script page injection task | wxt-content-script | partial / handoff |
| N01 | Negative | 模糊 WXT 泛任务 | L0.wxt | partial |
| N02 | Negative | 强指定 target_pack=security-permissions 但 task 实际是 manifest 主分析 | security-permissions 或 failed | handoff / failed |
| N03 | Negative | artifact 缺失 | wxt-manifest | partial / failed |
| N04 | Negative | domain validator fail | wxt-manifest | partial / failed |
| H01 | Handoff | manifest 输出后 release 成为主问题 | wxt-manifest -> release-engineering | handoff |

## 三、每类 case 的最低断言

### G01：主成功路径
必须断言：
- route 到 `wxt-manifest`
- bundle 带上 `security-permissions` 与 `release-engineering`
- 生成 `manifest-review.md`
- 跑 `validator-core-output`
- 跑 `validator-domain-wxt-manifest`
- activation result 合法

### G02：content-script 分支路径
必须断言：
- route 到 `wxt-content-script`
- 若需要则带 `L2.wxt.content-script.page-injection`
- 可生成轻量 artifact 或 handoff
- 不误入 manifest 主路径

### N01：模糊任务
必须断言：
- 不强行深路由
- 不默认扫描宿主
- 返回补充上下文建议

### N02：显式 target 与 task 冲突
必须断言：
- 系统能给出冲突解释
- 不因模糊 hints 覆盖显式 target_pack
- 如 target_pack 明显不成立，可返回 failed 或建议新 request

### N03：artifact 缺失
必须断言：
- core validator fail 或至少 warn
- activation result 不能是 completed

### N04：domain validator fail
必须断言：
- activation result 不能是 completed
- summary 明确说明缺少哪些领域结论

### H01：正常交接
必须断言：
- handoff bundle 有 `from_pack` / `to_pack` / `reason`
- confirmed assumptions 与 open questions 存在
- 当前 pack 不继续吞并主问题

## 四、建议的测试文件组织
```text
tests/
├─ fixtures/
│  ├─ manifest-basic/
│  ├─ manifest-store-submission/
│  ├─ content-script-page-injection/
│  └─ insufficient-context/

│  ├─ manifest-route.spec.*

│  └─ insufficient-context-route.spec.*

   ├─ core-output.spec.*
   └─ wxt-manifest-domain.spec.*
```

## 五、闭环验收清单

### 结构层
- activation request 可进入系统
- route 可选主 pack
- bundle 可最小生成
- artifact 可生成
- validator 可运行
- activation result 可返回

### 语义层
- 不默认扫描宿主
- 不把仓库当业务主仓
- 不把 wxt-manifest 扩成 all-in-one
- 模糊任务诚实退回 L0 / partial

### 结果层
- 至少一条主任务链跑通
- 至少一个 core validator 生效
- 至少一个 domain validator 生效
- 至少有一条正常 handoff
- 至少有一条 partial / fail 负例

## 六、Phase 1 闭环成立的判定
只有同时满足以下条件，才可认定第一阶段闭环成立：
1. 固定主任务可重复跑通
2. 主 artifact 稳定可生成
3. 两个主 validators 稳定生效
4. activation result 状态不是凭感觉给出的
5. 上下文不足与 handoff 都有可验证路径

## 非目标
本文不定义：
- 第二领域线的测试矩阵
- 大量边缘 case
- 性能测试
第一阶段只关心 operating model 是否成立。
